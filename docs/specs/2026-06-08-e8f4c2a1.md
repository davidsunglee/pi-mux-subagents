# Centralize subagent runtime diagnostics and warning delivery

Source: IDEA-e8f4c2a1

## Goal

Centralize user-visible runtime diagnostics and warning delivery behind a small,
typed diagnostics abstraction so that delivery is selected by runtime context
instead of decided locally at each call site. Warning *text* stays stable, but a
single dispatcher routes each warning to the right channel(s): interactive
TUI sessions get `ctx.ui.notify(message, "warning")` (no raw stderr to corrupt the
active editor), headless/CLI/non-UI contexts preserve stderr observability, and
orchestration/`subagent` tool calls additively surface caller-relevant warnings as
narrow structured metadata in the tool result. The end state is that no
warning-producing site chooses between `ui.notify`, structured metadata, and
`process.stderr.write` on its own — it emits a typed diagnostic and the dispatcher
decides.

## Context

Warning delivery in `pi-mux-subagents` is partially centralized already, via an
ad-hoc injectable-callback pattern that this work formalizes:

- **Warning producers** are pure functions that take an injectable
  `write: (msg: string) => void` callback defaulting to `process.stderr.write`:
  `warnGuardedPolicyUnsupported` (`src/launch/launch-spec.ts:787`),
  `warnClaudeSkillsDropped` (`src/index.ts:957`), and
  `warnCodexUnsupportedFeatures` (`src/index.ts:968`). The latter two emit the
  `ignoring skills=…` / `ignoring tools=…` and Codex `system-prompt: replace is
  not representable` strings.
- **The existing router** is `emitRuntimeWarning(message)` (`src/index.ts:487`):
  if `latestCtx?.hasUI` it calls `ui.notify(message, "warning")`, else it writes
  stderr. `latestCtx` is the module-level `ExtensionContext` captured at
  `session_start` (`src/index.ts:485`).
- **Wiring today**: the pane/orchestration launch path injects
  `emitRuntimeWarning` as the `write` callback (`src/index.ts:1045,1064,1134`), so
  those warnings UI-route when a UI exists; the headless path
  (`src/backends/headless.ts:552,739`) uses the stderr default.
- **Other raw-stderr sites**: headless child-runner process diagnostics
  (stream-format drift, transcript-archival failures) at
  `src/backends/headless.ts:678,690,893,902,938`; the invalid `PI_SUBAGENT_MODE`
  warning in `selectBackend()` (`src/backends/select.ts:18`), a pure zero-arg
  env-reading function with no `ctx` handle that already dedupes via
  `warnedInvalidValues`; and the `pi-mux-detect` CLI (`src/bin/pi-mux-detect.ts:94,108`).
- **The structured channel already exists**: tool handlers return
  `{ content, details }` and `details` is the structured payload the parent model
  sees (`src/tools/tool-handlers.ts`). The bare `subagent` tool and the
  orchestration tools (`subagent_run_serial` / `subagent_run_parallel`) both use
  this shape. There is no `warnings` field on `details` today, and
  `OrchestrationResult` / `OrchestratedTaskResult` (`src/orchestration/types.ts`)
  carry no warning field.
- **Existing tests** assert warning wording byte-for-byte and current routing:
  `test/orchestration/claude-skills-warning.test.ts`,
  `test/orchestration/codex-skills-warning.test.ts`,
  `test/orchestration/pane-warning-ui-routing.test.ts`,
  `test/orchestration/select-backend.test.ts`, and
  `test/integration/headless-claude-skills-warning.test.ts`.

## Requirements

- Introduce a typed `Diagnostic` record and a single central dispatcher
  (`emitDiagnostic(diagnostic, context)` or equivalent) that owns *all* routing
  between the UI channel, stderr, and the structured collector. The record
  carries, at minimum: the message text, an `audience` designation, and a stable
  identifier/code. Exact field names and signature are a planner decision.
- Model `audience` as a per-diagnostic property over the set `{ human, structured }`:
  `human` selects UI-or-stderr delivery; `structured` selects the tool-result
  collector. A warning may carry both. The dispatcher resolves each audience
  independently against the context.
- Context-aware `human`-audience delivery (preserving today's `emitRuntimeWarning`
  behavior): when an active UI is present, deliver via
  `ctx.ui.notify(message, "warning")`; otherwise fall back to stderr.
- `structured`-audience delivery is **conditional on a collector being present in
  the context**. When no collector exists (e.g. a `/subagent` slash-command pane
  launch, or any non-tool-call path), the `structured` portion is a silent no-op —
  not an error — and a `{ human, structured }` warning still delivers to the human
  channel normally.
- Migrate the three launch/orchestration warning producers
  (`warnGuardedPolicyUnsupported`, `warnClaudeSkillsDropped`,
  `warnCodexUnsupportedFeatures`) to emit typed `Diagnostic` records through the
  dispatcher, replacing the injected `write` callback + `emitRuntimeWarning`
  indirection.
- Migrate the invalid `PI_SUBAGENT_MODE` warning in `selectBackend()` to route
  through the dispatcher **without** changing its zero-arg signature or threading
  `ctx` into it. Provide an ambient default context that resolves the UI from the
  existing module-level `latestCtx`, so pure/zero-arg callers join the abstraction
  via one import. `selectBackend()`'s existing per-value dedupe is preserved.
- At the tool-call entry points (bare `subagent` tool, `subagent_run_serial`,
  `subagent_run_parallel`), create a per-call collector, thread it into the
  dispatch context, and surface collected `structured`-audience warnings
  *additively* in the tool result `details` as a narrow `warnings` field
  (per-task for orchestration tools). Human-channel delivery is unchanged.
- Initial per-diagnostic audience assignment:
  - `audience: { human, structured }` — `ignoring skills=…`, `ignoring tools=…`,
    `execution-policy=guarded requested but backend has no guarded mode`, and
    Codex `system-prompt: replace is not representable`.
  - `audience: { human }` only — transcript-archival failures, stream-format /
    `system/init` / `thread.started` drift warnings, Claude session-file-not-found,
    and invalid `PI_SUBAGENT_MODE`.
- Preserve every existing warning string byte-for-byte; the same string delivered
  to the human channel is the string mirrored into `details.warnings`.

## Constraints

- Existing warning wording is frozen: all current byte-for-byte wording tests must
  continue to pass unchanged, and the human-facing text must not change when a
  warning also surfaces as structured metadata.
- The structured-metadata path is *additive only*. It must not remove, replace, or
  gate any human-channel delivery, and it must not alter the existing `details`
  payload shape beyond adding the narrow `warnings` field.
- Delivery decisions live in exactly one place (the dispatcher). No call site may
  branch locally between `ui.notify`, structured metadata, and `process.stderr.write`.
- The ambient default context must reuse the existing module-level `latestCtx`
  rather than introducing a second source of UI-context truth.
- Do not rework general child-process stderr capture, spawned-tool process error
  handling, or CLI warning wording.
- Peer/runtime contract is unchanged: no new dependencies, no new user-facing
  configuration or verbosity flags.

## Approach

**Chosen approach:** A typed `Diagnostic` record plus one central dispatcher
(option (b) from the architecture round). Producers construct a typed `Diagnostic`
value carrying the message, a stable code, and an `audience` set, then hand it to a
single `emitDiagnostic(diagnostic, context)`. The dispatcher inspects the context
(is a UI present? is a structured collector present?) together with the record's
`audience`, and routes accordingly: `human` → `ctx.ui.notify` when UI present else
stderr; `structured` → push into the collector when one is present, else no-op. An
ambient default context resolves the UI from the existing module-level `latestCtx`,
so pure/zero-arg callers such as `selectBackend()` route through the same dispatcher
without a signature change. Tool-call entry points create a per-call collector,
thread it as context, and read the collected warnings into `details.warnings`.

**Why this over alternatives:** It most directly encodes the two decisions locked
during Q&A. Per-diagnostic audience becomes a first-class field on the record
rather than something split across sink composition, and "no site decides delivery
locally" becomes literally true — there is exactly one dispatcher that every
producer and `selectBackend()` route through identically. Centralizing the
UI-vs-stderr-vs-structured decision in one function is the cleanest fit for the
"right long-term design" priority the user set.

**Considered and rejected:**

- **Typed sink objects (Strategy/DI):** a `DiagnosticsSink` interface with concrete
  `UiSink` / `StderrSink` / `CollectingSink` implementations and a composite for
  "both". Rejected because it splits the audience decision between "which sink you
  compose" and the diagnostic itself, and the `{ human, structured }` case requires
  explicit sink composition at each call site — distributing exactly the delivery
  decision we want centralized.
- **Typed-callback formalization:** keep the existing injectable `write` callback,
  just type it (`DiagnosticsSink = (message, opts?) => void`) with canned
  factories. Rejected as the narrow option: audience becomes a second-class `opts`
  field on a function rather than a modeled concept, and the function-shaped
  contract tends to drift back toward ad-hoc local decisions.

## Acceptance Criteria

- A pane/TUI warning-path test proves a launch warning is delivered via
  `ctx.ui.notify(message, "warning")` and does **not** write raw stderr when UI
  notification is available (preserves the existing
  `pane-warning-ui-routing.test.ts` guarantee through the new dispatcher).
- A non-UI pane-launch test proves the same warning text falls back to stderr
  unchanged.
- A headless backend warning test proves current stderr behavior is preserved for
  child-runner process diagnostics.
- A CLI test proves `pi-mux-detect` diagnostics still write to stderr unchanged.
- A `selectBackend()` test proves the invalid `PI_SUBAGENT_MODE` warning routes
  through the dispatcher — landing on stderr when no UI is present, and on
  `ui.notify` when an ambient UI context is present — with dedupe intact and the
  zero-arg signature unchanged.
- An orchestration / `subagent` tool-result test proves caller-relevant warnings
  (e.g. `ignoring skills=…`) appear additively in `details.warnings` (per-task for
  orchestration tools) **without** changing the human-facing warning text or the
  rest of the `details` payload.
- A test proves a `{ human, structured }` warning emitted on a non-tool pane launch
  (no collector in context) delivers to the human channel only and does not error.
- All existing byte-for-byte warning-wording tests pass unchanged.
- A `human`-only warning (e.g. transcript-archival failure) is proven to never
  appear in `details.warnings`.
- A new warning-producing call site can be wired by constructing a `Diagnostic` and
  calling the dispatcher, with no local branching between UI, structured, and
  stderr delivery.

## Non-Goals

- Not force-migrating the pure process-level stderr sites — the headless
  child-runner diagnostics (`src/backends/headless.ts:678,690,893,902,938`) and the
  `pi-mux-detect` CLI — to call the dispatcher. They run where no UI and no
  collector exist, so routing them would be zero behavior change and pure churn.
  The dispatcher is *capable* of stderr/`human`-only delivery so these contexts are
  first-class and may adopt it later, but this change leaves those call sites as
  raw `process.stderr.write`.
- Not introducing any `model`-only (structured-without-human) warning. The audience
  taxonomy supports it, but no existing warning is assigned that audience now.
- Not changing any warning wording, the `details` schema beyond the additive
  `warnings` field, or the public tool/parameter surface.
- Not reworking spawned-tool process-error handling or general child-process stderr
  capture.
- Not adding diagnostics verbosity levels, filtering, persistence, or
  user-configurable channels.

## Open Questions

- Element shape of `details.warnings`: plain `string[]` (simplest, matches the
  byte-for-byte human text) versus structured `{ code, message }[]` (lets the
  parent model branch on a stable code). Leaning plain strings for v1 with the
  stable code retained internally on the `Diagnostic`; the planner can decide
  whether to expose the code in the tool payload.
