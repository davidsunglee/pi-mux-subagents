# Make Codex Pane Completion Use a Backend-Specific subagent_done Protocol

Source: IDEA-f6eac473

## Goal

Codex-backed pane subagents can leave their panes open under `auto-exit: true` because the Codex launch path reuses Claude's completion wording — variants of "your FINAL assistant message should summarize what you accomplished, then call `subagent_done`." In Codex, emitting a final answer can be terminal, so the trailing tool call never happens and the pane never closes. This work replaces the ad-hoc, Claude-flavored completion text on the Codex pane path with a backend-specific, **tool-first** completion protocol delivered through a typed seam, while keeping the parent-facing lifecycle contract (`subagent_done` → pane closes, summary returned) identical across pi, Claude, and Codex. The design lifts every completion decision that is currently made implicitly at each launch site into one exhaustive seam, so Codex — and any future backend — is structurally forced to declare a correct completion protocol rather than silently inheriting Claude's.

## Context

The pane launch path lives in `src/index.ts` and `src/launch/launch-spec.ts`; both pane and headless backends consume the same `ResolvedLaunchSpec`.

- **Task assembly today (`launch-spec.ts`, `resolveLaunchSpec`).** `spec.fullTask` is a single concatenated string: `roleBlock` (agent identity, when not in the system prompt) + `modeHint` + `params.task` + `summaryInstruction`. The two middle/last segments are **completion-protocol wording** wearing a "shared" costume:
  - `modeHint` — autonomous: "Complete your task autonomously."; interactive: "Complete your task. When finished, call the subagent_done tool. The user can interact with you at any time."
  - `summaryInstruction` — autonomous: "Your FINAL assistant message should summarize what you accomplished."; interactive: "Your FINAL assistant message (before calling subagent_done or before the user exits) should summarize what you accomplished."
  - `spec.claudeTaskBody` is the same minus `roleBlock` (Claude identity arrives via `--append-system-prompt`).
  - `spec.claudeCompletionAddendum` is populated **only** for `effectiveCli === "claude"` from `buildClaudeCompletionAddendum(autoExit)`, which returns final-message-then-`subagent_done` wording.
- **The bug (`index.ts`, Codex pane branch ~L1124–1157).** The Codex path builds `promptBody = ${spec.fullTask}\n\n${buildClaudeCompletionAddendum(spec.autoExit)}` (commented "CLI-neutral text"). So the Codex prompt carries final-message-first completion wording from **two** sources: the appended Claude addendum *and* the `summaryInstruction` baked into `fullTask`.
- **Shared MCP `subagent_done` tool (`src/claude-plugin/mcp/server.ts` → `server.js`).** A single stdio tool with an optional `message` arg; its `TOOL_DESCRIPTION` says "Your final assistant message before this call should summarize…" (final-answer-first). It backs **both** Claude (registered via `--plugin-dir`) and Codex (registered via `-c mcp_servers.*`), reading the CLI-neutral sentinel env var `PI_SUBAGENT_DONE_SENTINEL` (falls back to `PI_CLAUDE_SENTINEL`).
- **Codex MCP registration (`src/backends/codex-mcp.ts`, `buildCodexMcpOverrideArgs`).** Emits the `-c mcp_servers.pi_subagent.*` overrides plus `tools.subagent_done.approval_mode="approve"` so the unattended pane can call the tool without a human approval.
- **Watcher (`index.ts` ~L1676–1710).** Summary precedence is sentinel content → transcript last assistant message → pane screen scrape → exit-code string. This defensive recovery is independent of the model-facing instructions.
- **pi's native completion (`src/tools/subagent-done.ts`).** A separate pi-extension `subagent_done` tool (no `message` arg; summary = last assistant message) plus `caller_ping` and an `.exit`/auto-exit sidecar lifecycle. This is pi's native path and is out of scope.
- **Existing test.** `test/orchestration/pane-claude-completion-addendum.test.ts` asserts the Claude addendum wording and its fold into `--append-system-prompt`.
- **Sibling work.** `IDEA-9f7c852d` covers Codex agent-identity/system-prompt delivery; it proposes a Codex-specific task assembly (`codexTaskBody`). It must compose with this spec, not duplicate or hide completion instructions. `IDEA-4af1b8bf` (closed) added the Codex backend and the CLI-neutral sentinel.

## Requirements

1. **Decompose the task assembly** so the CLI-neutral core (agent identity block + task payload) is a clearly-marked structural unit, free of completion-protocol wording. The `modeHint`/`summaryInstruction` completion text is no longer baked into a shared string that every backend inherits. `fullTask`/`claudeTaskBody` survive as **derived** outputs so existing consumers stay behavior-stable.
2. **Introduce a typed pane completion-protocol seam** with explicit per-backend variants — not a generic parameterized addendum generator. All three pane backends (pi, Claude, Codex) resolve their completion behavior through this seam; there is no default/fallthrough path by which a backend silently inherits another's completion semantics.
3. **The seam is total over two dimensions, both enforced at compile time:**
   - *backend* — adding a pane CLI without a completion variant is a build error;
   - *execution mode* — each variant must supply **both** the autonomous (`auto-exit: true`) and interactive (`auto-exit: false`) forms; omitting either is a build error. A future third mode must likewise break every backend's variant until handled.
4. **Each backend variant declares, as mandatory slots:**
   - *terminality* — whether emitting a final answer ends the model's turn for that CLI. The shared composer **derives** the ordering posture (tool-first vs final-message-then-tool) from this declared fact; the ordering sentence is not hand-authored per backend.
   - *delivery channel* — where completion instructions are physically delivered (e.g. system-prompt vs task-prompt vs native extension).
   - *summary carrier* — whether the parent's summary arrives in-band via `subagent_done(message=…)` or out-of-band via the trailing assistant message.
   - *per-mode instruction text* — the autonomous and interactive wording.
5. **A single shared composer** assembles each pane backend's launch prompt from the neutral core plus that backend's completion contribution in one canonical order. Backends do not branch on mode, nor slice/override another backend's strings, at the call site.
6. **Codex pane variant.** Declares final-answer terminality (→ tool-first ordering), task-prompt delivery, and in-band `subagent_done(message=…)` summary carrier. Its instruction text, in both modes, states that: (a) a final answer is **not** the completion signal in Codex; (b) Codex must call `subagent_done` with the final summary in the `message` argument **before any terminal final answer**; (c) Codex should emit no further output after the tool call. The resulting Codex pane prompt contains **no** final-message-first / "summarize in your FINAL assistant message" wording from any source.
7. **Codex mode parity.** Interactive (`auto-exit: false`) Codex panes retain the "you may ask the user clarifying questions" framing while still carrying the tool-first rule; autonomous Codex panes carry the "complete without asking the user" framing plus the tool-first rule.
8. **Isolate the Claude builder.** Codex command/prompt construction no longer calls the Claude completion builder. The Claude-specific completion text is reachable only as the Claude seam variant; no other backend path can invoke it or inherit its wording.
9. **Rewrite the shared MCP `subagent_done` description** (`src/claude-plugin/mcp/server.*`) to be backend-neutral and tool-first: drop the "your final assistant message before this call should summarize…" framing and instead direct the model to provide its summary in the `message` argument. The rewrite must stay correct for Claude (whose watcher already reads sentinel/`message` content before falling back to the transcript).
10. **`subagent_done(message=…)` is the required Codex pane success path.** Codex is not instructed to rely on visible marker output. The watcher's existing defensive recovery (sentinel content → transcript last assistant message → screen scrape → exit-code string) remains unchanged.
11. **Preserve** pi and Claude observable behavior and the parent-facing lifecycle contract, and preserve all headless behavior. pi's native completion extension (`src/tools/subagent-done.ts`) is not modified.
12. **Tests:**
    - launcher/unit coverage asserting the Codex pane prompt/instructions contain the tool-first rule (call `subagent_done(message=…)` before any final answer) and do **not** contain Claude-only final-message-first phrasing, in **both** autonomous and interactive modes;
    - seam-level/exhaustiveness coverage proving that a new pane backend (and a new mode) cannot compile/launch without an explicit completion variant — i.e. no silent inheritance of Claude semantics;
    - existing pi and Claude pane-completion tests continue to pass with behavior unchanged.

## Constraints

- The parent-facing lifecycle is uniform and unchanged: every pane backend completes through `subagent_done`, the pane closes, and the summary is returned. Only the model-facing instructions differ by backend.
- The seam is **pane-scoped**. Headless completion (process exit + terminal JSONL event + `--output-last-message`) is not instruction-driven. The task-assembly decomposition must preserve all existing headless prompts/behavior, and pane completion instructions (e.g. "call `subagent_done`") must **not** be routed into headless Codex, which has no such tool.
- Do not modify pi's native `subagent-done.ts` extension tool, its tool surface, or its `.exit`/auto-exit lifecycle.
- Keep Codex MCP registration and auto-approval mechanics (`-c mcp_servers.*`, `tools.subagent_done.approval_mode="approve"` in `buildCodexMcpOverrideArgs`) in the Codex command-construction layer; they are launch mechanics, not part of the instruction seam.
- The MCP `subagent_done` description is shared by Claude and Codex panes; the rewrite must remain correct for Claude.
- Do not change the watcher's completion-detection mechanism or its summary-source precedence, beyond what naturally follows from the neutral MCP description.
- The decomposition is behavior-preserving for every existing prompt consumer (pi pane, Claude pane, both headless paths); the **only** intended prompt change is the Codex pane becoming tool-first.

## Approach

**Chosen approach:** Structured task parts + a typed, exhaustive completion-protocol seam, joined by one shared composer.

- The resolver stops emitting completion wording inside a shared `fullTask` string. The CLI-neutral core (identity block + task payload) becomes marked structural data; `fullTask`/`claudeTaskBody` survive as derived outputs so pi, Claude, and headless stay behavior-stable.
- Completion wording lives **only** behind the seam — a single per-backend completion-protocol record, total over backend × {autonomous, interactive}, where each backend declares terminality (→ derived ordering), delivery channel, summary carrier, and per-mode instruction text. Both dimensions are compile-time obligations; a new backend or a new mode breaks the build until handled.
- A shared composer assembles `neutral-core + backend-completion-contribution` in one canonical order. Codex's variant is tool-first and additive, so Codex never slices or overrides another backend's strings, and its prompt is free of final-message-first wording by construction.

**Why this over alternatives:**

- It delivers a genuine, structurally-marked common portion (not a convention), and makes "interactive vs autonomous" and "is a final answer terminal?" mandatory slots the next backend author is handed — not decisions they must rediscover.
- Deriving ordering from declared terminality makes "Codex gets Claude's message-first wording" structurally impossible, not merely discouraged.
- It composes with `IDEA-9f7c852d`: once identity is a first-class structured part, the Codex composer can route it through whatever channel Codex needs.

**Considered and rejected:**

- *(b) One centralized assembler parameterized by an injected completion-protocol object* — exhaustive and low-churn, but the common-vs-contributed boundary stays logical (protocol slots inside a monolithic string) rather than a structurally exposed core; weaker against the "clean decomposition" goal.
- *(c) Append-only completion block over a stripped core* — keeps the core completion-free, but placement/ordering is decided at each call site and drifts across backends; the shared composer in (a) removes that risk.
- *Minimal addendum-swap (replace only the trailing Codex addendum, leave the shared summary wording)* — the shared `summaryInstruction` would still tell Codex to summarize in a final message, re-introducing the same race the idea targets.

## Acceptance Criteria

- A Codex pane launch with `auto-exit: true` instructs the model to call `subagent_done(message=…)` before any final answer, and the launch prompt contains no "summarize in your FINAL assistant message / final message then call `subagent_done`" wording.
- A Codex pane launch with `auto-exit: false` still tells the model it may ask the user clarifying questions, and also carries the tool-first rule.
- The Codex pane path does not call the Claude completion builder; the Claude completion wording is reachable only as the Claude seam variant.
- Adding a hypothetical new pane backend, or a new execution mode, fails to compile/build until an explicit completion variant covering every mode is supplied — verified by a seam-level/exhaustiveness test.
- The shared MCP `subagent_done` description no longer implies a final assistant message should precede the call and instead directs the summary into the `message` argument; Claude pane completion behavior is unchanged.
- pi and Claude pane launches and all headless launches are behavior-unchanged from the parent-facing lifecycle perspective; existing pi/Claude pane-completion tests pass.
- A regression test inspects the Codex launch prompt/argv in both modes and fails if Claude-only final-message-first wording reappears on the Codex path.
- Manual smoke test: a simple `auto-exit: true` Codex pane subagent calls `subagent_done(message=…)` and the pane closes without a manual `exit`.

## Non-Goals

- Changing headless completion semantics for any CLI, or adopting the pane completion-protocol seam in the headless path.
- Modifying pi's native `subagent-done.ts` extension, its tool surface, or its auto-exit/`.exit` lifecycle.
- Changing the watcher's completion-detection mechanism or its sentinel → transcript → screen → exit summary precedence (beyond the neutral MCP-description wording).
- Moving Codex MCP registration / auto-approval mechanics into the seam.
- Codex agent-identity / system-prompt delivery (`system-prompt: append|replace`) — that is `IDEA-9f7c852d`; this spec only ensures the two compose (identity becomes a structured part the Codex composer can consume).
- Rewriting pi-flow workflow prompts (remains the pi-flow project's concern).
- Renaming `src/claude-plugin/` (retained deliberately per `IDEA-4af1b8bf`).

## Open Questions

- For the Claude pane variant, the planner has latitude in whether completion text stays split across the task body (`modeHint`/`summaryInstruction`) and the system-prompt addendum exactly as today, or is consolidated into the seam's single Claude variant. Either is acceptable provided Claude's observable behavior and lifecycle are unchanged and the no-silent-inherit (total-seam) property holds.
