# Deliver agent identity to Codex subagents when system-prompt append is used

Source: IDEA-9f7c852d

## Goal

Codex-backed subagents lose their agent identity (the agent body) whenever the agent
declares `system-prompt: append` or `system-prompt: replace`. In that mode the launcher
routes identity through a "system-prompt channel" that Claude and pi have but Codex does
not, so the agent body is silently dropped from the Codex task — leaving the model to guess
the task contract (the pi-flow `test-runner` artifact format is the triggering case). This
work makes identity delivery a first-class, per-backend concern — symmetric with the existing
completion-protocol seam — so a backend that has no system-prompt channel (Codex) always
receives the composed identity in its task body, while Claude and pi keep their current
behavior unchanged.

## Context

What exists today:

- `resolveLaunchSpec` (`src/launch/launch-spec.ts`) composes `identity` from the trimmed agent
  body plus the per-call `systemPrompt`. It sets `identityInSystemPrompt = !!(systemPromptMode
  && identity)`. When that is true, `roleBlock = ""`, so identity is omitted from both
  `fullTask` and `neutralCore`; identity is instead expected to reach the backend via a
  `--system-prompt` / `--append-system-prompt` flag materialized by `writeSystemPromptArtifact`.
- **Claude** is clean: it consumes `claudeTaskBody` (which never carries identity) and always
  routes identity through the system-prompt flag — one backend, one identity channel.
- **pi** is a hybrid: identity rides `fullTask`'s `roleBlock` when no mode is set, else routes
  through the flag (`writeSystemPromptArtifact`, pane `src/index.ts`, headless
  `src/backends/headless.ts`).
- **Codex** has no system-prompt flag — `buildCodexExecArgs` / `buildCodexPaneCmdParts`
  (`src/backends/codex-stream.ts`) expose no equivalent. The pane path feeds `spec.neutralCore`
  into `composePanePrompt` (`src/index.ts` ~1131); the headless path writes `spec.fullTask` to
  stdin (`src/backends/headless.ts` ~779). Both drop identity when `identityInSystemPrompt` is
  true. The headless comment asserting "fullTask carries the identity roleBlock" only holds
  when the mode is unset — that is the bug.
- The completion-protocol seam (`src/launch/pane-completion-protocol.ts`) already models
  completion delivery as a total, compile-time-checked, per-backend table
  (`PANE_COMPLETION_PROTOCOLS`, keyed backend × mode, with a `deliveryChannel`) whose explicit
  goal is "no silent inheritance." Identity delivery is **not** modeled with that rigor; this
  bug is precisely a silent-inheritance failure in identity delivery — Codex inherited the
  system-prompt-channel assumption that only holds for backends that actually have one. Codex
  already declares `deliveryChannel: "task-prompt"` for completion, so identity/completion text
  riding the Codex prompt body is consistent with existing design.
- No in-repo agent declares `system-prompt: append|replace`; the affected agents (e.g. pi-flow
  `test-runner`) live downstream, so regression coverage must use synthetic agent fixtures,
  matching the temp-agent-file pattern already in `test/orchestration/launch-spec.test.ts` and
  `test/system-prompt-mode.test.ts`.
- `warnCodexUnsupportedFeatures` (`src/index.ts` ~968), called on both Codex paths, already
  warns when skills/tools are dropped on Codex — a natural home for a "mode not representable"
  warning.
- Downstream: pi-flow `IDEA-9dfc2062` ("Make Codex test-runner subagents produce valid
  artifacts") tracks the symptom; this fix likely resolves its central root-cause open question
  (Codex ignoring artifact-format instructions vs never receiving them). Codex tool-first
  completion wording is tracked in `IDEA-f6eac473`.

## Requirements

- Model identity delivery as a first-class, per-backend capability parallel to the
  completion-protocol seam: each pane/headless backend declares whether it has a system-prompt
  channel for identity. Both the task-body composition and the system-prompt-flag emission
  derive from that declaration rather than from the bare `identityInSystemPrompt` boolean.
- Codex declares "no system-prompt channel": the composed identity is always included in the
  Codex task body, on both the pane path and the headless path, regardless of `systemPromptMode`.
- Claude and pi retain their current observable behavior; the new seam **encodes** (does not
  change) Claude's always-system-prompt routing and pi's hybrid routing. Identity is never
  duplicated in any backend's user task.
- `system-prompt: replace` on Codex delivers identity identically to `append` (additively, in
  the task body) and emits a one-time runtime warning (e.g. via the existing Codex
  unsupported-features warning) stating that exact base-instruction replacement is not
  representable on Codex and that identity was delivered additively.
- Identity delivery composes with the existing Codex completion seam: the Codex task prompt
  contains identity + task + the tool-first completion wording with no duplication and no
  omission of completion instructions (composes with `IDEA-f6eac473`).
- The "no silent inheritance" property holds for identity as it does for completion: a new
  pane/headless backend cannot be added without declaring its identity-delivery capability.
- Regression coverage:
  - a Codex pane launch for an agent declaring `system-prompt: append` with a non-empty body
    includes that body in the composed task prompt;
  - the Codex headless launch for the same agent includes that body in the stdin prompt;
  - a fixture modeled on pi-flow `test-runner` (artifact-format contract living in the agent
    body) shows the Codex prompt includes the required headers/markers, e.g. `COMMAND:`,
    `EXIT_CODE:`, `END_FAILING_IDENTIFIERS`, and `--- RAW RUN OUTPUT BELOW ---`;
  - `replace`-on-Codex emits the warning.

## Constraints

- Preserve existing Claude command construction: identity continues to reach Claude via
  `--system-prompt` / `--append-system-prompt`, never duplicated in the user task body.
- Preserve existing pi launch behavior (role-block-in-body when no mode; system-prompt artifact
  when a mode is set) and fork/lineage session semantics.
- Do not mutate user configuration or the working repo to deliver identity (no writing
  `AGENTS.md`, no persisting to `~/.codex/config.toml`); stay consistent with the existing
  per-launch `-c` override approach.
- Codex resume: identity travels with the task body on initial and resume sends alike —
  preserve the current "prompt written to stdin on every send" behavior; do not add
  resume-specific identity suppression as part of this work.
- Do not weaken pi-flow artifact parsers or rewrite the pi-flow `test-runner` artifact format
  (downstream, out of scope).

## Approach

**Chosen approach:** First-class identity-delivery seam. Identity delivery becomes a total,
per-backend capability modeled parallel to `PANE_COMPLETION_PROTOCOLS`: each backend declares
whether it owns a system-prompt channel for identity. Backends with a channel (Claude always;
pi when a mode is set) route identity to the flag; a backend without one (Codex) always carries
identity in its task body. The body-composition logic and the flag-emission logic both derive
from this declaration instead of from the bare `identityInSystemPrompt` boolean, so Codex's
"no channel" row makes correct identity delivery fall out by construction for both the pane and
headless paths.

**Why this over alternatives:** The codebase already invested in exactly this seam pattern for
completion delivery and documents "no silent inheritance" as a first-class goal. This bug is a
silent-inheritance failure in identity delivery — Codex inherited a system-prompt-channel
assumption it does not satisfy. Making identity symmetric with completion both fixes this
instance and prevents the next channel-less backend from repeating it, while encoding Claude/pi
behavior unchanged.

**Considered and rejected:**

- Local fix at the two Codex launch sites — smallest diff, but identity delivery stays ad-hoc
  and the next channel-less backend can hit the same trap; leaves the completion/identity
  asymmetry in place.
- Explicit `codexTaskBody` field mirroring `claudeTaskBody` — explicit and low-risk, but still
  per-backend special-casing; a future channel-less backend needs its own field again, and the
  no-silent-inheritance property is not enforced.

## Acceptance Criteria

- A unit test proves a Codex pane launch for an agent declaring `system-prompt: append`
  (non-empty body) includes the agent body in the composed task prompt — it no longer drops it.
- Equivalent coverage proves the Codex headless launch for the same agent includes the body in
  the stdin prompt.
- A `test-runner`-modeled fixture shows the Codex prompt / launch arguments include the
  artifact-format contract from the body, including `COMMAND:`, `EXIT_CODE:`,
  `END_FAILING_IDENTIFIERS`, and `--- RAW RUN OUTPUT BELOW ---`.
- Requesting `system-prompt: replace` on Codex emits the one-time "not representable, delivered
  additively" warning and still includes identity in the body.
- The Codex task prompt contains identity, task, and completion wording together, with no
  duplicated and no missing completion instructions.
- A hypothetical new pane/headless backend that does not declare its identity-delivery
  capability fails to compile (parallel to the completion-seam totality guard).
- Existing Claude command-construction tests still pass and show no duplicated identity in the
  Claude user task.
- Existing pi launch behavior and the existing `launch-spec` / `system-prompt-mode` tests still
  pass unchanged.
- Manual (cross-repo) validation — a Codex-backed `test-runner` dispatch in pi-flow writes an
  artifact accepted by `pi-flow helper _shared/parse-test-runner-artifact` — is tracked in
  pi-flow `IDEA-9dfc2062` and is **not** an automated gate in this repo.

## Non-Goals

- True base-instruction replacement on Codex (`system-prompt: replace` exactly) — not
  representable; replace is best-effort additive delivery plus a warning.
- Changing Claude or pi identity-routing behavior; the seam encodes their current behavior.
- Adding skills/tools support on the Codex path (a separate, already-warned limitation).
- Weakening or changing pi-flow's `parse-test-runner-artifact` parser or the `test-runner`
  artifact format.
- Changing pi-flow model-dispatch policy to avoid Codex (a downstream workaround tracked in
  pi-flow `IDEA-9dfc2062`, not this fix).

## Open Questions

- Exact representation of the Codex identity-bearing body (a derived `codexTaskBody` field vs an
  inline composition driven by the seam) is left to planning; the seam should make the correct
  value fall out rather than relying on a hand-maintained field. (Idea Open Question 3.)
- Whether to re-inject identity on Codex `resume` or rely on the prior session's context; this
  spec defaults to preserving the current send-with-body behavior. If prior-session identity is
  sufficient, a future refinement could suppress re-injection on resume.
- This fix is expected to resolve pi-flow `IDEA-9dfc2062`'s root-cause open question (Codex
  ignoring artifact-format instructions vs never receiving them). Confirming that downstream —
  and deciding whether pi-flow still wants tool-assisted artifact generation as
  defense-in-depth — remains pi-flow's call.
