# Support Codex CLI for Subagent Execution

Source: IDEA-4af1b8bf

## Goal

Add the Codex CLI as a third subagent execution backend, selectable as `cli: "codex"`, with full parity to the existing Claude backend across both transports — headless (`codex exec`) and interactive mux-pane. Parity covers direct launches, serial/parallel orchestration, completion semantics, cancellation/error handling, model and thinking mapping, resume, and guarded-vs-unrestricted execution-policy behavior mapped onto Codex's sandbox/approval controls. The work preserves existing `pi` and `claude` behavior unchanged and never mutates the user's persistent Codex configuration.

## Context

The backend layer is two-tiered and CLI-agnostic at the resolution step, then splits by transport:

- **Resolution:** `resolveLaunchSpec()` (`src/launch/launch-spec.ts`) normalizes a launch into a `ResolvedLaunchSpec` — `effectiveCli` (`params.cli ?? agentDefs.cli ?? "pi"`), `effectiveModel`, `effectiveThinking`, `effectiveExecutionPolicy`, cwd/session placement, identity/system-prompt, etc. This is shared by both transports.
- **Transports:** `makeHeadlessBackend()` (`src/backends/headless.ts`) spawns child processes and parses JSON streams; `makePaneBackend()` (`src/backends/pane.ts`) delegates to `launchSubagent()`/`watchSubagent()` (`src/index.ts`) which drive mux panes. `selectBackend()` (`src/backends/select.ts`) chooses transport (pane/headless) from `PI_SUBAGENT_MODE` + mux detection, independent of CLI.
- **CLI routing:** both transports branch on CLI — `if (spec.effectiveCli === "claude") { … } else { /* pi */ }` — in `launchSubagent()` (pane) and inside `makeHeadlessBackend()`'s runner dispatch (`runClaudeHeadless` vs `runPiHeadless`). Codex adds a third branch in each.
- **Command builders:** `buildClaudeCmdParts()` (`src/index.ts`, pane) and `buildClaudeHeadlessArgs()` (`src/backends/claude-stream.ts`, headless); pi argv is built inline in both. Codex needs analogous builders.
- **Completion semantics:** headless completion is a JSON-stream terminal event plus process close — Claude's terminal `result` event (`parseClaudeResult`), pi's `message_end` + `stopReason`. Pane completion is `pollForExit()` (`src/mux/index.ts`) watching a `.exit` sidecar, a screen `__SUBAGENT_DONE_<n>__` marker, and — for Claude — a `sentinelFile` written by an MCP `subagent_done` tool. The Claude MCP server (`src/claude-plugin/mcp/server.ts`) is a small stdio server auto-loaded via `--plugin-dir`; it writes the sentinel path from an env var (`PI_CLAUDE_SENTINEL`). `pollForExit()` already accepts a `sentinelFile` option, so detection is transport-ready for a new CLI.
- **Execution policy:** `guarded` (default) / `unrestricted` is resolved CLI-agnostically in `launch-spec.ts`. Claude maps it to permission modes (`--permission-mode auto` vs `--dangerously-skip-permissions`/`bypassPermissions`); pi currently runs unrestricted; `warnGuardedPolicyUnsupported()` emits a warning when guarded is explicitly requested for a backend with no guarded mode. The README "Execution policy → Future backend mappings" table already names Codex's *intended* mapping, which this spec refines and promotes to "implemented."
- **Model/thinking:** there is a single `model:` frontmatter field / `model` tool param expressed in pi's provider-qualified syntax (`provider/model[:thinking]`). pi passes it through (folding thinking as a `:suffix`); Claude strips the leading `provider/` prefix via `projectModelForClaude()` and maps thinking to `--effort`. There is no model alias table — the provider prefix is load-bearing for pi routing but advisory (stripped) for non-pi CLIs.
- **Resume:** `resumeSessionId` → Claude `--resume <id>`; pi resumes via its session file.
- **Tools/skills:** `PI_TO_CLAUDE_TOOLS` maps tool names; `warnClaudeSkillsDropped()` warns and ignores pi `skills` on the Claude path.
- **Tests:** `test/integration/harness.ts` defines `TEST_MODEL` (default `anthropic/claude-haiku-4-5`, overridable via `PI_TEST_MODEL`) and `SLOW_LANE_OPT_IN` (`PI_RUN_SLOW === "1"`). Real-CLI integration tests self-skip via a `which <cli>` availability probe (e.g. `CLAUDE_AVAILABLE` gating `headless-claude-smoke.test.ts`).
- **Codex CLI surfaces** (from the idea and local `codex-cli 0.136.0`): interactive `codex`; non-interactive `codex exec` and `codex exec resume <session-id>`; JSONL via `--json`; final-message capture via `--output-last-message <file>`; workspace via `--cd/-C`; model via `--model/-m`; config overrides via `-c key=value`; MCP server configuration; safety via `--sandbox`, `approval_policy` / `--ask-for-approval`, and `--dangerously-bypass-approvals-and-sandbox`. The maintained binary is `codex` on `PATH`. Per-launch `-c` overrides registering an `[mcp_servers.*]` block (`command` + `args` array + `env` table) have been verified to work on the installed Codex (see Approach).

## Requirements

1. Accept `cli: "codex"` everywhere `pi`/`claude` are accepted: the bare `subagent` tool, `subagent_run_serial` / `subagent_run_parallel` steps, and agent frontmatter `cli`. `effectiveCli` resolution extends to `codex`.
2. Discover only the `codex` binary on `PATH`. A missing binary fails with a clear, actionable error (mirroring the existing "claude CLI not found on PATH"). Do not add alternate CLI selectors, a separate binary-path override, or a `codex-cli` executable dependency.
3. Add a Codex command builder and a Codex routing branch to **both** transports (headless and pane), alongside the existing pi/claude branches.
4. **Headless:** run `codex exec` with `--json`, `--output-last-message <file>`, and `--cd <cwd>`, delivering the prompt via stdin or a direct argument as appropriate. Treat process exit plus the final-message file / JSONL stream as completion. Produce a `BackendResult` carrying `exitCode`, `finalMessage`, `transcript`, `transcriptPath`, `usage` (when present in JSONL), and `sessionId` (when Codex emits a session/thread id).
5. **Pane:** run interactive `codex` in the mux pane with an explicit `subagent_done` completion path provided by a Codex MCP server registered via per-launch `codex -c` config overrides (`mcp_servers.<name>.command` / `.args` / `.env`), mirroring the Claude sentinel behavior. Reuse `pollForExit()`'s existing `sentinelFile` plumbing for detection and the **same** inner stdio `subagent_done` server that backs the Claude path (`src/claude-plugin/mcp/`), generalized to read a **CLI-neutral sentinel env var** so one server serves both CLIs (Claude's observable behavior unchanged — only the internal env-var name moves, updated on both the server and `buildClaudeCmdParts`). The `src/claude-plugin/` directory keeps its name and Claude-plugin packaging; Codex registers the server directly via `-c mcp_servers`, not `--plugin-dir`. No persistent Codex config or state is written (see Approach).
6. **Execution-policy mapping** (per-mode):
   - *guarded headless:* `workspace-write` sandbox with non-interactive approvals — `approval_policy = "never"` applied via a `-c` config override (because non-interactive `codex exec` cannot answer `on-request` prompts).
   - *guarded pane:* `workspace-write` sandbox with `on-request` approvals (interactive, so a human can answer).
   - *unrestricted (both):* `--dangerously-bypass-approvals-and-sandbox`.
   - Codex must be recognized as having a guarded mode so no spurious "guarded unsupported" warning fires for Codex launches.
7. **Model mapping:** project the pi provider-qualified model by stripping the leading `provider/` prefix to a bare name passed to Codex `--model` (reusing the established Claude prefix-strip contract); bare names pass through unchanged. Do not build a Codex model-alias table. Provider and auth selection remain the responsibility of Codex config/profile.
8. **Thinking mapping:** map supported `thinking` values to `-c model_reasoning_effort=<minimal|low|medium|high|xhigh>`; omit unsupported values rather than passing invalid flags.
9. **Resume:** support `resumeSessionId` via `codex exec resume <session-id>` once a session/thread id is reliably parsed from JSONL output. When the installed Codex does not emit a stable id, resume is unsupported for that run and fails with a deterministic error plus a one-line stderr warning — never a silent new session. Parse defensively; do **not** gate on a Codex CLI version (mirroring the Claude stream-format-drift handling).
10. Warn and ignore pi-only `skills` and tool allowlists on the Codex path (mirroring `warnClaudeSkillsDropped`), **except** the internal MCP completion tool required for pane completion.
11. Preserve existing `pi` and `claude` behavior unchanged, including Claude plugin/sentinel compatibility. (The sentinel env-var rename in Requirement 5 changes an internal name only; Claude's observable pane-completion behavior is identical.)
12. **Docs:** update README / API docs and examples for Codex usage and safety-model differences. Promote Codex from "future" to "implemented" in the execution-policy mapping table with the per-mode rows from Requirement 6, document that Codex `guarded` is sandbox-enforced but **not** classifier-backed (unlike Claude's `--permission-mode auto`), and document the never-mutate-persistent-config guarantee.
13. **Test harness model defaults:** when `PI_TEST_MODEL` is unset, select a default model by CLI — `pi → openai-codex/gpt-5.4-mini`, `claude → claude-haiku-4-5`, `codex → gpt-5.4-mini`. When `PI_TEST_MODEL` is set, it remains a single global override applied regardless of CLI.
14. **Tests:** add unit/contract coverage for command construction, per-mode policy mapping, missing-binary error, JSONL parsing, result shaping, and routing; add gated real Codex integration tests for headless and mux-pane/orchestration flows, self-skipping via a `which codex` probe and opting into the slow lane via `PI_RUN_SLOW`. Existing pi and Claude unit and integration tests continue to pass unchanged.

## Constraints

- Never mutate the user's persistent `~/.codex/config.toml` (or other persistent Codex state) when injecting the MCP server or applying policy/model/thinking settings.
- `cli: "codex"` is the only public selector; no `codex-cli` executable dependency and no binary-path override.
- No runtime CLI-version detection or gating for Codex; assume the installed-or-later Codex version at runtime.
- Switching pi's default test model from `anthropic/claude-haiku-4-5` to `openai-codex/gpt-5.4-mini` means the default pi integration lane requires `openai-codex` auth instead of Anthropic auth — any CI or contributor running pi tests without `PI_TEST_MODEL` set will need Codex credentials.
- Codex async orchestration cannot use pi's `caller_ping` block/ping path (that tool exists only inside pi children); Codex tasks run to terminal like Claude tasks.
- Provider and auth selection stay Codex's responsibility (config/profile); no provider-alias mapping is introduced.

## Approach

**Chosen approach:** Register the pane-mode Codex `subagent_done` MCP server via **per-launch `codex -c` config overrides** — `mcp_servers.<name>.command`, `.args` (array), and `.env` (table) passed as `-c` flags on each interactive `codex` invocation. This is the **sole** injection mechanism: it writes nothing to disk, so the user's persistent `~/.codex` config is untouched by construction. The server itself is the existing stdio `subagent_done` server under `src/claude-plugin/mcp/` (kept where it is — see below), generalized to read a CLI-neutral sentinel env var. Completion *detection* reuses `pollForExit()`'s `sentinelFile` option. The headless path, command builders, routing branches, and policy/model/thinking mapping slot mechanically into the existing pi/claude structure.

**Viability confirmed against the installed Codex.** Probed with `codex mcp list` under `-c 'mcp_servers.pi_probe.command="echo"' -c 'mcp_servers.pi_probe.args=["hello","world"]' -c 'mcp_servers.pi_probe.env={PI_PROBE="1"}'`: Codex listed `pi_probe` as **enabled** with the command, args array, and env table all parsed from the overrides. No temporary config layer is required.

**Directory naming.** The `src/claude-plugin/` directory keeps its name. It is a Claude-plugin-compliant package (manifest + `--plugin-dir` layout), and the name accurately communicates that shape; renaming it to something "neutral" would obscure real information. Codex does not consume the plugin packaging — it registers the inner `mcp/server.js` directly via `-c mcp_servers` — so cross-CLI reuse needs only the sentinel env var to be neutral, not a directory rename.

**Why this over alternatives:** the `-c` mechanism is the least-invasive option (a flag, not a disk mutation — the structural analog of Claude's ephemeral `--plugin-dir`), honors the hard "never mutate persistent config" constraint by construction, and keeps exactly one code path to build, document, and test ("the simplest thing that works, and only that thing").

**Considered and rejected:**

- **Ephemeral `CODEX_HOME` overlay** (a temp config dir layering the MCP block over copies of the user's `config.toml` + `auth.json`, set only for the spawned process and deleted after the run) — this was the designated fallback had `-c` been unable to express the nested block; unnecessary now that `-c` injection is verified, and it would add temp-dir lifecycle plus auth/config copying for no benefit.
- **Process-exit / screen-sentinel only (no MCP)** — forfeits the in-band `subagent_done` signal for interactive (non-auto-exit) Codex panes, breaking the v1 interactive-parity requirement.
- **Layered runtime `-c` + `CODEX_HOME` fallback** — explicitly rejected: two code paths to maintain and test for no user-visible benefit; the mechanism is fixed (a single shipped path), not selected per run.
- **Temp profile via `--profile` / `-c profile`** — Codex profiles live in `config.toml` and do not define MCP servers (MCP config is global), so a profile alone cannot inject the server.
- **Renaming `src/claude-plugin/` to a CLI-neutral directory** — rejected: the name accurately denotes a Claude-plugin-compliant package, Codex doesn't consume the packaging, and the only contract that must be CLI-neutral (the sentinel env var) is neutralized without a rename.

## Acceptance Criteria

- `cli: "codex"` launches a Codex-backed subagent from every public entry point that accepts `pi`/`claude` (bare tool, serial, parallel, and agent frontmatter).
- A missing `codex` on `PATH` fails with a clear, actionable error.
- Guarded and unrestricted launches emit the documented per-mode Codex flags/`-c` overrides: guarded headless → `workspace-write` + `approval_policy = "never"`; guarded pane → `workspace-write` + `on-request`; unrestricted → `--dangerously-bypass-approvals-and-sandbox`. No spurious guarded-unsupported warning fires for Codex.
- Headless Codex runs return `exitCode`, `finalMessage`, `transcript`, `transcriptPath`, `usage` (when present in JSONL), and a session/thread id when Codex emits one.
- Pane Codex runs can ask questions, accept user input, call `subagent_done` (via the `-c`-injected MCP server), and report completion to the parent without relying on process exit alone — and the user's persistent `~/.codex/config.toml` is unchanged after the run.
- Serial and parallel orchestration run Codex tasks and report terminal states consistently with the pi and Claude backends.
- A gated smoke/integration test runs a unique-marker task on a low-cost Codex model (e.g. `gpt-5.4-mini`), self-skipping when `codex` is absent and opting into the slow lane via `PI_RUN_SLOW`.
- Resume works when Codex emits stable ids; when unavailable on the installed Codex version, the unsupported path yields a deterministic error, with test coverage.
- Model `openai-codex/gpt-5.4-mini` projects to `--model gpt-5.4-mini`; thinking maps to `-c model_reasoning_effort=<value>`, and unsupported thinking values are omitted.
- With `PI_TEST_MODEL` unset, the harness applies the per-CLI default models (`pi → openai-codex/gpt-5.4-mini`, `claude → claude-haiku-4-5`, `codex → gpt-5.4-mini`); with `PI_TEST_MODEL` set, it overrides globally.
- Existing pi and Claude unit and integration tests pass unchanged, including Claude pane-completion (sentinel) behavior under the renamed internal env var.

## Non-Goals

- Pane-mode MCP injection via persistent-config mutation, or any persistent Codex state changes.
- A temporary `CODEX_HOME` overlay or any alternate/fallback injection path — the single shipped mechanism is per-launch `-c` config overrides (verified viable on the installed Codex).
- Renaming the `src/claude-plugin/` directory — it is retained deliberately because the name accurately denotes a Claude-plugin-compliant package (manifest + `--plugin-dir` layout). Cross-CLI reuse of its inner `subagent_done` MCP server is achieved via a CLI-neutral sentinel env var and direct `-c mcp_servers` registration from the Codex path, not a rename.
- Codex provider-alias mapping or provider/auth management (Codex config/profile owns it).
- pi-style `caller_ping` / block-resume orchestration for Codex (Codex tasks run to terminal like Claude).
- Skills support and pi-style tool allowlists for Codex (warn-and-ignore in v1, except the internal MCP completion tool).
- Alternate Codex CLI selectors, a `codex` binary-path override, or a `codex-cli` dependency.
- Per-CLI `PI_TEST_MODEL` override env vars (the design is a single global override plus per-CLI defaults only).
- A runtime Codex-version gate.

## Open Questions

- **Exact JSONL event names/shape** for the session/thread id and usage fields on the installed Codex version — to be confirmed during implementation against real `codex exec --json` output. Defensive parsing is required regardless (per Requirement 9).
