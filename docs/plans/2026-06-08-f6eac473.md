# Plan: Make Codex Pane Completion Use a Backend-Specific `subagent_done` Protocol

**Source:** `IDEA-f6eac473`
**Spec:** `docs/specs/2026-06-08-f6eac473.md`

## Goal

Codex-backed pane subagents can leave their panes open under `auto-exit: true` because the Codex launch path reuses Claude's final-message-first completion wording (`buildClaudeCompletionAddendum`) on top of the `modeHint`/`summaryInstruction` baked into the shared `fullTask`. In Codex a final answer can end the turn, so the trailing `subagent_done` call never happens. This work replaces the Claude-flavored Codex completion text with a **backend-specific, tool-first** completion protocol delivered through a typed, compile-time-exhaustive seam, joined by one shared composer — while keeping the parent-facing lifecycle (`subagent_done` → pane closes → summary returned) identical across pi, Claude, and Codex.

## Architecture summary

A new seam module `src/launch/pane-completion-protocol.ts` defines a typed `PaneCompletionProtocol` record that is **total over `PaneBackend × PaneExecutionMode`** (pi/claude/codex × autonomous/interactive), enforced at compile time by a mapped-type table plus union-exhaustiveness guard constants. Each variant declares four mandatory slots — `terminality`, `deliveryChannel`, `summaryCarrier`, and per-mode `framing` text. A shared `composePanePrompt({ neutralCore, protocol })` assembles a backend's launch prompt from a **neutral core** (identity + task, free of completion wording) plus the backend's completion contribution, in one canonical order; the ordering *sentence* (tool-first vs final-message-then-tool) is **derived** from `terminality`, never hand-authored per backend.

`resolveLaunchSpec` gains a `neutralCore` field (identity + task, no completion wording) and re-sources `claudeCompletionAddendum` from the seam via the composer. `fullTask` and `claudeTaskBody` survive **byte-identical** as derived outputs so pi pane, Claude pane, and both headless paths are behavior-stable. Only the **Codex pane** prompt changes: it now uses `composePanePrompt(spec.neutralCore, codex[mode])`, which contains the tool-first rule and no final-message-first wording. The free function `buildClaudeCompletionAddendum` is deleted so Claude's wording is reachable **only** as the Claude seam variant. The shared MCP `subagent_done` tool description (`src/claude-plugin/mcp/server.ts` → `server.js`) is rewritten to be backend-neutral/tool-first.

The seam is **pane-scoped**: headless Codex still writes the unchanged `spec.fullTask` to stdin; no pane completion instructions are routed into headless. pi's native `subagent-done.ts` extension and the watcher's summary precedence are untouched.

## Reference: byte-exact strings that MUST be preserved

These are the current outputs that downstream consumers (pi pane, Claude pane, both headless paths) depend on. The decomposition must keep them byte-identical.

- `fullTask` (non-fork) = `` `${roleBlock}\n\n${modeHint}\n\n${params.task}\n\n${summaryInstruction}` `` where `roleBlock = identity && !identityInSystemPrompt ? `\n\n${identity}` : ""`.
- `claudeTaskBody` (non-fork) = `` `${modeHint}\n\n${params.task}\n\n${summaryInstruction}` ``.
- `modeHint`:
  - autonomous: `Complete your task autonomously.`
  - interactive: `Complete your task. When finished, call the subagent_done tool. The user can interact with you at any time.`
- `summaryInstruction`:
  - autonomous: `Your FINAL assistant message should summarize what you accomplished.`
  - interactive: `Your FINAL assistant message (before calling subagent_done or before the user exits) should summarize what you accomplished.`
- Claude completion addendum (current `buildClaudeCompletionAddendum`):
  - autonomous: `You are a one-shot subagent. Complete your task autonomously without asking the user questions. When finished, your FINAL assistant message should summarize what you accomplished, then call ` + "`subagent_done`" + ` to end the session.`
  - interactive: `You are an interactive subagent. The user can type into this pane at any time — feel free to ask clarifying questions as many times as needed. When the task is complete, your FINAL assistant message should summarize what you accomplished, then call ` + "`subagent_done`" + ` to end the session.` (note the em-dash `—`, U+2014)

The Claude seam variant's `framing + " " + <derived ordering sentence>` must reproduce these two strings exactly (see Task 1).

  ~~~ts
  /**
   * Pane completion-protocol seam.
   *
   * Every pane CLI backend (pi, Claude, Codex) declares how it signals task
   * completion to the model. This seam is total over backend × execution-mode
   * and the totality is enforced at COMPILE TIME (see PANE_COMPLETION_PROTOCOLS
   * and the exhaustiveness guards below): adding a backend or a mode without an
   * explicit variant is a `tsc` error, so no backend can silently inherit
   * another's completion semantics.
   *
   * The shared composer (composePanePrompt) assembles a pane backend's launch
   * prompt from a neutral core (identity + task, free of completion wording)
   * plus the backend's completion contribution, in one canonical order. The
   * ordering SENTENCE (tool-first vs final-message-then-tool) is DERIVED from
   * each backend's declared `terminality` — it is never hand-authored per
   * backend.
   */

  export type PaneBackend = "pi" | "claude" | "codex";
  export type PaneExecutionMode = "autonomous" | "interactive";

  /**
   * Whether emitting a final assistant answer can END the model's turn.
   * - "final-answer-terminal": a final answer may end the turn (Codex) →
   *   TOOL-FIRST ordering posture.
   * - "final-answer-nonterminal": the model keeps control after a final answer
   *   (Claude, pi) → FINAL-MESSAGE-THEN-TOOL ordering posture.
   */
  export type Terminality = "final-answer-terminal" | "final-answer-nonterminal";

  export type CompletionDeliveryChannel =
    | "system-prompt" // Claude — folded into --append-system-prompt
    | "task-prompt" // Codex — appended to the pane task prompt
    | "native-extension"; // pi — driven by the subagent-done.ts extension

  export type SummaryCarrier =
    | "in-band-message-arg" // subagent_done(message=…)
    | "out-of-band-final-message"; // trailing assistant message

  export type OrderingPosture = "tool-first" | "final-message-then-tool";

  export interface PaneCompletionProtocol {
    backend: PaneBackend;
    mode: PaneExecutionMode;
    terminality: Terminality;
    deliveryChannel: CompletionDeliveryChannel;
    summaryCarrier: SummaryCarrier;
    /**
     * Per-mode framing text, MINUS the ordering sentence. The composer appends
     * the derived ordering sentence (resolveOrderingSentence). End this string
     * WITHOUT a trailing space; the composer joins with a single space.
     */
    framing: string;
  }

  /** Ordering posture is a pure function of terminality. */
  export function deriveOrderingPosture(t: Terminality): OrderingPosture {
    return t === "final-answer-terminal" ? "tool-first" : "final-message-then-tool";
  }

  /**
   * The ordering sentence is derived from posture + summary carrier; it is NOT
   * hand-authored per backend. Both currently-used combinations are covered.
   */
  export function resolveOrderingSentence(
    posture: OrderingPosture,
    _carrier: SummaryCarrier,
  ): string {
    if (posture === "tool-first") {
      // in-band carrier (Codex)
      return (
        "In Codex a final answer can end your turn, so a final answer is NOT the " +
        "completion signal: call `subagent_done(message=…)` with your full summary " +
        "in the `message` argument before sending any final answer, and send no " +
        "further output after that call."
      );
    }
    // final-message-then-tool, out-of-band (Claude, pi) — byte-identical to the
    // legacy buildClaudeCompletionAddendum tail.
    return (
      "your FINAL assistant message should summarize what you accomplished, then " +
      "call `subagent_done` to end the session."
    );
  }

  /** Build the full per-mode instruction (framing + derived ordering sentence). */
  export function buildInstructionText(p: PaneCompletionProtocol): string {
    const sentence = resolveOrderingSentence(
      deriveOrderingPosture(p.terminality),
      p.summaryCarrier,
    );
    return p.framing ? `${p.framing} ${sentence}` : sentence;
  }

  export interface ComposedPanePrompt {
    /** Task-prompt text for the pane CLI's prompt argument. */
    taskPrompt: string;
    /** Completion text for the system prompt, or null when N/A. */
    systemPromptCompletion: string | null;
  }

  function joinCore(core: string, trailing: string): string {
    return core.trim().length > 0 ? `${core}\n\n${trailing}` : trailing;
  }

  /**
   * Single shared composer. Joins the neutral core with the backend's
   * completion contribution in one canonical order (core, then trailing
   * completion for the task-prompt channel). Callers pass a neutral core and a
   * protocol and read the field for their delivery channel — no per-backend
   * branching at the call site.
   */
  export function composePanePrompt(input: {
    neutralCore: string;
    protocol: PaneCompletionProtocol;
  }): ComposedPanePrompt {
    const { neutralCore, protocol } = input;
    const instruction = buildInstructionText(protocol);
    switch (protocol.deliveryChannel) {
      case "task-prompt":
        return { taskPrompt: joinCore(neutralCore, instruction), systemPromptCompletion: null };
      case "system-prompt":
        return { taskPrompt: neutralCore, systemPromptCompletion: instruction };
      case "native-extension":
        // pi delivers completion via its native extension; the composer adds no
        // prompt text and passes the core through unchanged.
        return { taskPrompt: neutralCore, systemPromptCompletion: null };
    }
  }
  ~~~

## Risk Assessment

- **Byte-identity of pi/Claude pane and both headless prompts.** The hard constraint is that only the Codex pane prompt changes. Mitigation: `fullTask`, `claudeTaskBody`, `modeHint`, and `summaryInstruction` are left untouched in `resolveLaunchSpec`; the Claude seam `framing` strings are split so `framing + " " + <derived ordering sentence>` reproduces the legacy addendum byte-for-byte (locked by strict-equality tests in Tasks 1 and 2 against the Reference strings); pi's direct task arg flows through the composer's `native-extension` passthrough, which returns the core unchanged (locked by a `deep-equal` passthrough test). The em-dash `—` (U+2014) in the Claude interactive framing must be copied exactly.

- **Interpretation of requirement 5 ("a single shared composer assembles each pane backend's launch prompt").** All three backends resolve their completion through the seam and the shared `composePanePrompt`: Codex via `taskPrompt` (task-prompt channel), Claude via `systemPromptCompletion` (system-prompt channel, invoked in `resolveLaunchSpec` to populate `claudeCompletionAddendum`), and pi via the native-extension passthrough. The canonical order is "core, then trailing completion" for the task-prompt channel. pi and Claude retain their legacy in-body mode framing inside their derived `fullTask`/`claudeTaskBody` (explicitly permitted: the spec's `## Approach` keeps `fullTask`/`claudeTaskBody` as derived outputs for behavior stability, and the Open Questions grant latitude for the Claude variant). The genuine neutral core (`spec.neutralCore`, free of completion wording) exists as the marked structural unit and is what the Codex path consumes — so the "shared string every backend inherits" is broken specifically for Codex, which is the bug target.

- **`modeHint`/`summaryInstruction` still present in `fullTask` (consumed by headless Codex).** This is intentional and behavior-preserving: headless completion is process-exit driven and out of scope; headless Codex already receives that wording today and must continue to (constraint: preserve all headless behavior; do not route the new pane seam wording into headless). The seam is pane-scoped and only consumed at pane launch sites, so headless is structurally untouched.

- **Composition with `IDEA-9f7c852d` (Codex identity delivery).** `spec.neutralCore` mirrors `fullTask`'s identity handling exactly: it includes the `roleBlock` only when `!identityInSystemPrompt`, and omits identity when an agent sets `system-prompt: append|replace` (the current behavior). This plan therefore does **not** change Codex identity delivery — it only changes completion wording — leaving `IDEA-9f7c852d` free to route identity into the Codex composer's core later. (Spec said the two must compose; this plan does Codex completion only and keeps identity behavior byte-stable.)

- **`server.js` drift.** The committed runtime artifact is regenerated by `npm run build:plugin`; the description test reads both `server.ts` and `server.js` to catch source/artifact drift, and Task 5 verifies a clean tree after committing the regenerated `server.js`.

- **Compile-time exhaustiveness depends on `tsc` running over tests.** The root `tsconfig.json` includes `test/**`, so the `@ts-expect-error` exhaustiveness assertions are validated by `npm run typecheck`. If the include scope ever narrows, those assertions would silently stop being checked — the runtime totality test (iterating `ALL_PANE_BACKENDS × ALL_PANE_EXECUTION_MODES`) plus the mapped-type table remain as a second line of defense.

- **`ban-ts-comment` / unused-var lint.** `@ts-expect-error` directives MUST include a description (enforced by `tseslint.configs.recommended`); guard constants MUST be `_`-prefixed (ESLint `no-unused-vars` ignores `^_`). Both are specified explicitly in Task 1; `npm run lint` in Task 5 is the backstop.
