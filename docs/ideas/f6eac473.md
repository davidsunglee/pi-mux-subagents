## Context

Codex pane subagents in pi-flow currently can run with the `pi_subagent.subagent_done` MCP server installed, but they can still leave panes open under `auto-exit: true`. Diagnostics showed that Codex can call the MCP tool when directly instructed, so the issue is not MCP registration or approval configuration.

The failure is caused by backend-specific completion semantics. The Codex launch path currently reuses Claude-oriented completion wording from `buildClaudeCompletionAddendum(spec.autoExit)`, including variants of "your FINAL assistant message should summarize what you accomplished, then call `subagent_done`." In Codex, emitting a final answer can be terminal, so the later MCP call never happens.

Claude, pi, and Codex should keep the same parent-facing lifecycle contract: a pane subagent reports completion through `subagent_done` so the parent can close the pane and return a summary. The model-facing instructions, however, must differ by backend because the CLIs interpret final answers and tool calls differently. Claude can continue to use final-message-then-tool wording, pi can continue to rely on its native extension lifecycle, and Codex must receive tool-first completion instructions.

This follows the closed Codex-backend implementation idea `IDEA-4af1b8bf` and narrows the remaining work to backend-specific completion protocol and command construction. It should also compose with `IDEA-9f7c852d`, which covers Codex prompt/identity delivery.

## Goal

Make Codex-backed pane subagents close reliably when `auto-exit: true` by ensuring Codex receives completion instructions that call the MCP `subagent_done` tool with the completion message before any terminal final answer, while preserving existing Claude and pi behavior.

The design should make accidental reuse of Claude completion semantics on Codex difficult to reintroduce.

## Scope

- Introduce a narrow typed completion-protocol seam for pane launches, with explicit backend variants rather than a parameterized completion-addendum generator.
- Keep independent backend-specific instruction builders behind that seam:
  - Claude: completion instructions remain delivered through the Claude system-prompt path and may retain final-message-then-`subagent_done` wording.
  - Codex: completion instructions are delivered through the Codex pane prompt path and must require `subagent_done(message=...)` before any final answer.
  - pi: existing native extension lifecycle remains unchanged.
- Split Codex command/prompt construction from the Claude addendum path so Codex cannot call `buildClaudeCompletionAddendum(...)` directly or inherit Claude-only wording.
- Add a Codex-specific completion addendum for pane launches stating that:
  - a final answer is not the completion signal in Codex;
  - Codex must call MCP `subagent_done` with the final summary in the `message` argument before any terminal final answer;
  - Codex should emit no further output after the tool call.
- Revise the shared MCP `subagent_done` tool description to be backend-neutral and tool-first, avoiding wording such as "final assistant message before this call" that encourages final-answer-first behavior.
- Treat `subagent_done(message=...)` as the required Codex success path. Existing screen-scrape or exit-code summary fallback may remain as defensive recovery in the watcher, but Codex should not be instructed to rely on visible marker output.
- Add launcher/unit coverage asserting that the Codex prompt/addendum contains the tool-first completion rule and does not contain Claude-only final-answer-first phrasing.
- Add seam-level or exhaustiveness coverage so new pane backends must choose an explicit completion protocol instead of silently sharing Claude semantics.

Out of scope: rewriting pi-flow workflow prompts; that remains in the pi-flow project idea.

## Acceptance Sketch

- Codex pane launch uses a backend-specific completion-protocol path rather than the Claude completion addendum verbatim.
- The completion-protocol seam has explicit backend variants and does not use a generic parameterized addendum generator.
- The Codex addendum explicitly says to call `subagent_done(message=...)` before any final answer and not to send a terminal final answer alone.
- The MCP `subagent_done` description is backend-neutral/tool-first and no longer implies that a final assistant message should precede the tool call.
- Claude and pi launch behavior remain unchanged from the parent-facing lifecycle perspective.
- A regression test inspects the Codex launch prompt/argv and fails if Claude-only final-answer-first wording is reintroduced.
- Manual smoke test: a simple Codex pane subagent with `auto-exit: true` calls `subagent_done(message=...)` and closes without manual `exit`.
