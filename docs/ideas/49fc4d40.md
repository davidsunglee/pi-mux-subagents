## Context
Pi 0.79.0 added project trust gating for project-local settings, resources, packages, extensions, and skills, with `--approve` / `--no-approve` one-run controls. Pi 0.79.1 added `defaultProjectTrust` and `ctx.isProjectTrusted()` so extensions can observe effective trust decisions.

pi-mux-subagents already treats child-launch trust as an operational concern for other backends: Codex launches receive a per-launch trusted-project override for the cwd, and Claude launches avoid workspace-trust stalls via headless `-p` behavior or pane `CLAUDE_CODE_SANDBOXED=1`. Those trust-prompt bypasses are separate from `executionPolicy`: guarded still keeps backend permission/sandbox controls engaged, while unrestricted bypasses them.

Pi-backed children do not yet have equivalent explicit handling. They may prompt for project trust in pane mode or skip project-local resources in non-interactive mode depending on Pi defaults. At the same time, this extension performs its own project-local `.pi/agents` discovery, which must not bypass Pi's parent-side trust gate.

## Goal
Align Pi-backed subagent startup with existing Claude/Codex operational behavior: child launches should not stall on Pi project-trust prompts or silently miss project-local resources solely because they are running as subagents. Keep this separate from `executionPolicy`, which remains the autonomy/sandbox policy.

Also ensure untrusted project-local configuration cannot bootstrap itself into privileged child behavior through this extension's own agent discovery or frontmatter resolution.

## Scope
- Bump the minimum supported Pi peer/runtime version to `>=0.79.1` so the extension can rely on `ctx.isProjectTrusted()`.
- For Pi-backed child launches, add a one-run Pi project-trust override using `--approve` where the framework starts a child Pi process on the user's behalf, for both pane and headless launch paths.
- Do not add a public cross-backend `projectTrust` API yet. Backend support is uneven, and this change is about preserving current non-blocking subagent launch semantics rather than exposing a new user-facing policy dimension.
- Keep `executionPolicy` unchanged:
  - `guarded` continues to mean the backend's safer autonomous mode where available.
  - `unrestricted` continues to mean permission/sandbox bypass where available.
  - Pi project-trust approval is a launch-time input-loading decision, not an unrestricted execution policy.
- Gate this extension's parent-side project-local `.pi/agents` discovery using `ctx.isProjectTrusted()` so untrusted repositories cannot supply agent frontmatter that affects launch behavior before trust is established.
- Treat project-local agent frontmatter as unable to request or derive approval/trust behavior unless the project is already trusted.
- Document backend-specific trust behavior for Pi, Claude, Codex, and future OpenCode support.

## Acceptance Sketch
- Package metadata requires Pi `>=0.79.1`.
- Headless Pi child command construction includes `--approve` and has tests proving the flag is present without writing persistent trust state.
- Pane Pi child command construction includes `--approve` and has tests proving the flag is present in the generated launch command.
- Claude and Codex documentation continues to describe their existing launch-time trust handling separately from `executionPolicy`.
- Project-local `.pi/agents` are ignored or skipped when `ctx.isProjectTrusted()` says the project is not trusted.
- Tests cover that untrusted project-local agent frontmatter cannot bootstrap `execution-policy: unrestricted`, `cwd`, or any trust-related behavior into automatic approval.
- Documentation explains that project trust controls project-local input loading, while execution policy controls backend autonomy/sandboxing.
- Existing guarded/unrestricted backend mappings remain unchanged except for the explicit Pi project-trust launch approval.
