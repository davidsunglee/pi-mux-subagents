## Context
Pi 0.79.0 added project trust gating for project-local settings, resources, packages, extensions, and skills, with `--approve` / `--no-approve` for non-interactive runs. Pi 0.79.1 added `defaultProjectTrust` and `ctx.isProjectTrusted()` so extensions can observe effective trust decisions.

pi-mux-subagents already exposes `executionPolicy: "guarded" | "unrestricted"` for backend autonomy. That policy currently maps to Claude/Codex permission or sandbox behavior, but does not explicitly model Pi project trust. Pane/headless Pi subagents may now encounter trust prompts, or headless children may skip project-local resources depending on Pi defaults.

## Goal
Respect Pi's new project trust model without confusing it with execution autonomy. Guarded launches should preserve Pi's native trust flow, while explicitly unrestricted trusted/sandboxed launches should have a way to avoid trust prompts or missing project-local resources.

## Scope
- Keep `executionPolicy` as the autonomy/sandbox policy: `guarded` or `unrestricted`.
- Add or derive a separate Pi project-trust launch behavior for Pi-backed children:
  - guarded: no trust override; let Pi prompt in pane mode and use `defaultProjectTrust` behavior in headless mode.
  - unrestricted: consider passing one-run `--approve` for Pi child launches so trusted/sandboxed unrestricted runs do not stall or silently omit project-local resources.
- Consider an explicit future knob such as `projectTrust: "inherit" | "approve" | "deny"` mapping to no flag / `--approve` / `--no-approve`.
- Ensure project-local `.pi/agents` discovery in this extension respects Pi trust, likely via `ctx.isProjectTrusted()` on Pi >= 0.79.1.
- Revisit dependency bounds or feature detection for Pi 0.79.0/0.79.1 APIs and flags.

## Acceptance Sketch
- Documentation explains that project trust controls project-local input loading, while execution policy controls backend autonomy.
- Pi-backed child command construction has tests for guarded vs unrestricted trust flag behavior.
- Guarded Pi pane launches do not force-approve project trust.
- Unrestricted Pi launches can opt into/derive one-run approval without writing persistent trust state.
- Project-local agent definitions cannot use untrusted repo config to bootstrap `execution-policy: unrestricted` into automatic project approval.
- Extension-local project agent discovery is gated by effective project trust where supported.

## Open Questions
- Should unrestricted imply `--approve` automatically, or should a separate `projectTrust` parameter be required for explicitness?
- If unrestricted is specified by project-local agent frontmatter, should it be ignored for approval derivation until the project is already trusted?
- Should headless guarded launches pass `--no-approve` for deterministic behavior, or inherit Pi's `defaultProjectTrust`?
- Should the package minimum peer dependency become Pi >= 0.79.1 to rely on `ctx.isProjectTrusted()`, or should older Pi versions remain supported with feature detection?
