## Context
During execute-plan for `docs/plans/2026-06-08-f6eac473.md` in `pi-mux-subagents`, Wave 1 Tasks 1 and 4 were implemented and then verified by fresh verifier subagents. Both verifier reports were substantively clear: they listed each acceptance criterion as passing, included command outputs, showed the relevant tests/typecheck/lint/build commands exiting successfully, and summarized an overall PASS. However, `pi-flow helper execute-plan/parse-verifier-report` returned `VERDICT: FAIL` because the reports were syntactically nonconforming: missing exact `[Criterion N] PASS` headers, missing an exact `VERDICT:` line, malformed `## Phase 1 Evidence` blocks, or `command:` fields that did not byte-match the extracted Verify recipe prose including backticks.

A retry produced even more explicit substantive PASS evidence, but still failed parser validation solely because the `command:` field paraphrased recipes (for example omitting backticks or including interpreted command text rather than the exact `Verify:` recipe string). This caused repeated verifier redispatches for what was effectively a formatting/parser issue, not a lack of implementation confidence. The user called this out as garbage syntactic validation and asked whether any existing ideas relax it; none were found in the current project ideas or pi-flow ideas.

The current execute-plan contract says malformed verifier output and protocol errors are treated as `VERDICT: FAIL`, and Step 11 requires `VERDICT: PASS` from `parse-verifier-report` before committing the wave. The acceptance-criteria-verification doc also says ad hoc / inferred-criteria mode is out of scope and recorded as a future-idea follow-up.

## Goal
Improve execute-plan verifier ergonomics so the orchestrator does not waste retries or re-run verification solely to repair report syntax when a verifier has already provided clear, auditable PASS evidence for every acceptance criterion.

## Scope
Explore and implement a safe relaxation or repair path for verifier protocol failures. Possible approaches include:

- Make `parse-verifier-report` more tolerant of common semantically equivalent shapes, such as `Criterion 1: PASS`, `## Overall Verdict\nPASS`, or evidence blocks with `Command:` / `Exit code:` capitalization variants.
- Separate substantive verdict parsing from Phase 1 command-provenance validation, so parser output can distinguish `semantic_pass_with_protocol_warnings` from true `FAIL`.
- Add an orchestrator-level protocol-repair step that asks a lightweight agent or helper to reformat an existing verifier report without re-running commands, preserving the original evidence verbatim.
- Add a user-confirmed override path when all criteria are explicitly PASS, all command exit codes are shown as successful, and the only parser errors are shape/format errors, not missing or failing evidence.
- Improve verifier prompting so command-style recipes include an explicit machine-copyable `command_field` separate from user-facing Verify prose, avoiding byte-equal matching against full natural-language recipe text with backticks and success-condition prose.

The solution must preserve the core safety boundary: the orchestrator should not independently inspect implementation files or invent acceptance evidence. Relaxation should only apply to verifier-produced evidence already present in the report, or to mechanical reformatting of that evidence.

## Acceptance Sketch
- A regression fixture based on the `f6eac473` Wave 1 verifier reports parses or routes without requiring verifier redispatch when all criteria are explicitly PASS and command outputs show success.
- True verifier failures still fail: missing criteria, failing command exit codes, absent evidence for a command-style criterion, or ambiguous PASS/FAIL wording must not be accepted silently.
- Parser/orchestrator output clearly distinguishes hard FAIL from protocol-only formatting defects.
- execute-plan documentation explains the allowed repair/override path and preserves the no-ad-hoc-orchestrator-verification boundary.
- Tests cover at least: exact legacy format, common human-readable PASS format, command-field paraphrase with preserved successful evidence, and genuine missing/failing evidence.

## Open Questions
- Should tolerance live entirely in `parse-verifier-report`, or should execute-plan add a separate protocol-repair/reformat sub-step?
- Is user confirmation required for accepting a semantic PASS with protocol warnings, or can it be automatic under tightly defined conditions?
- Should `Verify:` recipes be split into machine-command and human-success-condition fields during plan generation to avoid byte-equal matching against prose?
