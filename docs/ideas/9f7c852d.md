## Context

A pi-flow fastlane verification run exposed that Codex-backed `test-runner` subagents can receive a task prompt that references the agent definition while not receiving the agent definition itself. The run dispatched `test-runner` through Codex (`openai-codex/gpt-5.4-mini`) for `npm test` and Codex completed with the expected `TEST_RESULT_ARTIFACT:` marker, but wrote a malformed shortened artifact:

```text
PHASE: full-suite
FAILING_IDENTIFIERS:
NON_RECONCILABLE_FAILURES:
```

The pi-flow parser rejected it with:

```json
{"failure":"header_missing","artifact":"/Users/david/Code/pi-flow/docs/test-runs/IDEA-f8278ccd/full-suite.log","detail":"expected 'COMMAND', got 'FAILING_IDENTIFIERS:'"}
```

Systematic debugging of the actual Codex rollout showed the Codex prompt contained `test-runner-prompt.md` plus the auto-exit addendum, but did not contain the `test-runner.md` agent body or its `## Artifact Format` section. The prompt told Codex to use "your agent definition" and `## Artifact Format`; in the Codex session transcript, strings from that definition such as `COMMAND:`, `END_FAILING_IDENTIFIERS`, and `You are a test runner` were absent. Codex then wrote the shortened artifact via an `apply_patch` call, indicating the bad artifact was produced by the Codex child, not by the parent parser or watcher.

The likely pi-mux-subagents mechanism is:

- `test-runner.md` declares `system-prompt: append`.
- `launch-spec.ts` sets `identityInSystemPrompt = true` for that mode and omits the agent `roleBlock` from `spec.fullTask`.
- Claude launch paths can pass the identity via Claude's system-prompt mechanism.
- The Codex pane path in `src/index.ts` builds `promptBody = spec.fullTask + addendum` and passes that as the Codex task; it does not appear to pass an equivalent appended system prompt.
- The Codex headless path may have the same issue because it also writes `spec.fullTask` to stdin.

This is distinct from `IDEA-f6eac473`, which covers Codex-specific tool-first completion wording. That idea addresses whether Codex calls `subagent_done`; this idea addresses whether Codex receives the agent identity/body needed to perform the task correctly before completion.

Related downstream pi-flow artifacts:

- `IDEA-9dfc2062` in `/Users/david/Code/pi-flow`: Codex `test-runner` produces malformed artifacts.
- `IDEA-f8278ccd` in `/Users/david/Code/pi-flow`: pi-flow prompt wording changed toward tool-first completion.

## Goal

Make Codex-backed subagents receive the same effective agent identity/body and caller-supplied system prompt semantics that pi and Claude-backed subagents receive, including when an agent declares `system-prompt: append` or `system-prompt: replace`.

For pi-flow `test-runner`, Codex must receive the artifact-format contract from the agent body (or an equivalent Codex system/developer prompt) so it does not have to infer the artifact shape from an incomplete task prompt.

## Scope

- Audit Codex pane and headless launch paths for how they consume `ResolvedLaunchSpec` fields such as `identity`, `identityInSystemPrompt`, `systemPromptMode`, `fullTask`, and `claudeTaskBody`.
- Decide and implement Codex-specific prompt assembly, likely one of:
  - pass agent identity through a Codex-supported system/developer prompt mechanism if available; or
  - add a `codexTaskBody` / Codex prompt assembly path that embeds the agent identity into the task when Codex has no separate system-prompt channel.
- Preserve existing Claude behavior: Claude should continue using `--system-prompt` / `--append-system-prompt` semantics and should not receive duplicated role text in its user task.
- Preserve existing pi behavior and lineage/fork session semantics.
- Ensure `system-prompt: append` and `system-prompt: replace` have clear, tested semantics on Codex.
- Ensure Codex auto-exit addendum work from `IDEA-f6eac473` composes with the identity delivery fix rather than hiding or duplicating completion instructions.
- Add regression coverage that a Codex launch for an agent with frontmatter `system-prompt: append` and a non-empty body includes that body, or otherwise passes it through a verified Codex system/developer prompt channel.
- Add a regression fixture modeled on pi-flow `test-runner`: the generated Codex prompt or launch arguments should include `## Artifact Format` / required artifact headers when that contract lives in the agent body.
- Consider whether warning text should surface when Codex cannot support a requested system-prompt mode exactly.

Out of scope:

- Weakening pi-flow artifact parsers.
- Rewriting pi-flow test-runner artifact format.
- Changing pi-flow dispatch policy solely to avoid Codex; that is a downstream workaround if this runtime identity-delivery bug remains unresolved.

## Acceptance Sketch

- A unit test proves Codex pane launch no longer drops agent body text when the agent declares `system-prompt: append`.
- If Codex headless launch shares the same prompt assembly issue, equivalent coverage proves headless Codex also receives the effective identity.
- A fixture based on `test-runner` shows Codex receives the artifact-format contract, including required headers such as `COMMAND:`, `EXIT_CODE:`, `END_FAILING_IDENTIFIERS`, and `--- RAW RUN OUTPUT BELOW ---`.
- Existing Claude command-construction tests still pass and do not show duplicated agent identity in the user task.
- Existing pi launch behavior is unchanged.
- Manual smoke test: a Codex-backed `test-runner` dispatch for a passing command writes an artifact accepted by `pi-flow helper _shared/parse-test-runner-artifact`, or the remaining failure is shown to be unrelated to missing agent identity.

## Open Questions

- Does the current Codex CLI expose a reliable per-launch system/developer prompt channel, or should pi-mux-subagents embed identity directly into the Codex task body?
- Should Codex support `system-prompt: replace` exactly, or should it fail/warn if the semantics cannot be represented without changing base instructions?
- Should `ResolvedLaunchSpec` grow backend-specific task bodies (`claudeTaskBody`, `codexTaskBody`, etc.) to make this invariant explicit?
- Are there other agents with `system-prompt: append` whose Codex behavior is currently degraded because their body is dropped from `spec.fullTask`?
