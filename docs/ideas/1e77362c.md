# Goal

Allow subagents launched by `pi-mux-subagents` to run through Safehouse-backed Pi wrappers such as `pi-safe`, `pi-ios`, and `pi-ios-debug`, instead of always invoking literal `pi`.

This should cover pane, headless, and resume paths, and should preserve existing Pi/Claude semantics.

# Design direction

Do **not** model `pi-safe`, `pi-ios`, or `pi-ios-debug` as additional `cli` values.

Keep `cli` as the semantic runtime/protocol selector:

- `cli: pi` — Pi CLI, Pi session files, `subagent_done` / `caller_ping`, Pi stream/session behavior.
- `cli: claude` — Claude CLI, Claude session id, sentinel/plugin behavior.

Add a separate launcher/profile concept, e.g. `launchProfile`, `launcher`, or `sandboxProfile`, for how the selected runtime is invoked.

Example intent:

```yaml
cli: pi
launch-profile: ios
```

Resolved via user-owned config such as:

```json
{
  "launch": {
    "pi": {
      "defaultProfile": "safe",
      "profiles": {
        "bare": { "command": "pi" },
        "safe": { "command": "pi-safe" },
        "ios": { "command": "pi-ios" },
        "ios-debug": {
          "command": "pi-ios-debug",
          "requiresExplicitSelection": true
        }
      }
    },
    "claude": {
      "defaultProfile": "bare",
      "profiles": {
        "bare": { "command": "claude" }
      }
    }
  }
}
```

The exact config location/schema can be refined during implementation, but the core distinction should remain: `cli` selects protocol; launch profile selects wrapper/sandbox.

# Current behavior / problem

Subagent launch paths currently hard-code executables:

- Pane Pi launch builds command parts starting with literal `pi` in `pi-extension/subagents/index.ts`.
- Headless Pi launch uses `spawnImpl("pi", args, ...)` in `pi-extension/subagents/backends/headless.ts`.
- Pi resume path builds `pi --session ...` in `pi-extension/subagents/index.ts`.
- Claude paths similarly hard-code `claude`.

Launching the parent Pi via `pi-safe` is not sufficient for pane subagents: commands are injected into mux panes, whose shells are usually owned by the mux server rather than by the Safehouse-wrapped parent Pi process.

Headless children may inherit the parent sandbox if spawned as process children, but pane children need their own command to be Safehouse-backed.

Also, `cli: pi-safe` is a footgun today: unknown `cli` values fall onto the Pi path but still execute literal `pi`.

# Required work

## 1. Add launch-profile resolution

In `pi-extension/subagents/launch-spec.ts` and related types:

- Add a new field to `SubagentParams`, orchestration task schema, agent frontmatter parsing, and `ResolvedLaunchSpec`.
- Resolve effective launch profile from call params, agent defaults, config default, then fallback.
- Resolve the selected runtime command and any leading launcher args.
- Keep `effectiveCli` normalized to runtime/protocol (`pi` or `claude`).
- Reject unknown `cli` values rather than silently treating them as Pi.

Consider whether project-local agent frontmatter is allowed to request privileged profiles. At minimum, `ios-debug` should require explicit opt-in rather than being selected implicitly by untrusted repo config.

## 2. Replace hard-coded executables

Update all launch paths to use the resolved command/profile:

- Pane Pi initial launch.
- Headless Pi initial launch.
- Pi `subagent_resume({ sessionPath })` branch.
- Optionally prepare equivalent plumbing for Claude launchers later, but Pi/Safehouse support is the priority.

For wrapper commands, launcher/safehouse args must come before Pi args:

```bash
pi-ios --env-pass=... --session ... -e subagent-done.ts ...
```

because the wrappers route leading Safehouse flags to Safehouse and pass the rest to `pi`.

## 3. Preserve lifecycle environment under Safehouse

Safehouse uses a sanitized environment unless variables are explicitly passed. Pi subagent lifecycle currently depends on env vars including:

- `PI_SUBAGENT_NAME`
- `PI_SUBAGENT_AGENT`
- `PI_SUBAGENT_SESSION`
- `PI_SUBAGENT_ID`
- `PI_SUBAGENT_ACTIVITY_FILE`
- `PI_SUBAGENT_SURFACE`
- `PI_SUBAGENT_AUTO_EXIT`
- `PI_DENY_TOOLS`
- `PI_CODING_AGENT_DIR`

Either:

1. Update the `pi-safe` / `pi-ios` / `pi-ios-debug` wrappers to always pass a safe lifecycle allowlist through Safehouse, or
2. Have this project inject an appropriate leading `--env-pass=...` argument when invoking Safehouse profiles.

Without this, children may launch but lose identity, status reporting, caller ping behavior, deny-tool behavior, config-root propagation, or auto-exit behavior.

## 4. Handle cwd / workdir correctly

The wrappers use `safehouse --workdir="$PWD" -- pi "$@"`.

Initial pane/headless launches already run in the effective child cwd, which maps well to Safehouse workdir behavior.

Resume needs special attention: `subagent_resume({ sessionPath })` should run from the original session cwd or otherwise select the intended Safehouse workdir. Options:

- Parse the child session file header and `cd` to its `cwd` before invoking the wrapper.
- Store/resurface cwd in the resume path.
- Accept/document explicit resume cwd if needed.

Document that launching a child in a nested `cwd` narrows the Safehouse writable workdir unless extra directories are granted.

## 5. Ensure required files are accessible

Safehouse-backed subagents must be able to access:

- Child session file.
- Task/system-prompt artifact files.
- `subagent-done.ts` extension file.
- Project-local `.pi/agent` when applicable.
- Global Pi config/session state needed by the child.

Verify whether Safehouse's built-in Pi profile plus workdir covers this. If not, add explicit launcher args or documentation.

## 6. Tests and docs

Add regression tests for:

- Pane launch scripts use configured profile commands (`pi-safe`, `pi-ios`, etc.) instead of literal `pi`.
- Headless spawn uses the configured launcher command.
- Resume uses the configured launcher command and correct cwd/workdir.
- Unknown `cli` values fail loudly.
- Launcher args are ordered before Pi args.
- Safehouse lifecycle env vars are passed or explicitly preserved.
- `ios-debug` is never selected by default.

Update README/docs to explain:

- `cli` vs launch profile.
- How to configure `pi-safe`, `pi-ios`, and `pi-ios-debug`.
- Why pane subagents need explicit wrapper invocation even if the parent was launched via `pi-safe`.
- Safehouse network caveat and escalation paths from the sandboxing recommendations doc.

# Acceptance criteria

- A Pi subagent can be launched through `pi-safe` from a pane backend.
- A Pi subagent can be launched through `pi-safe` from the headless backend.
- `pi-ios` can be selected explicitly for Xcode/iOS tasks.
- `pi-ios-debug` is available only by explicit selection and is documented as privileged.
- `subagent_resume({ sessionPath })` uses the configured Pi launcher/profile.
- Existing `cli: pi` and `cli: claude` behavior remains compatible.
- Unknown `cli` values are rejected with a clear error.
- Tests cover command construction and launcher/profile resolution without requiring Safehouse to be installed.
