## Context

Pane-backed subagent launches can run inside Pi's interactive TUI. In that mode, direct writes to `process.stderr` can appear as raw terminal output and corrupt the active prompt editor or custom UI surfaces. One-off fixes have already routed specific pane launch warnings through `ctx.ui.notify(message, "warning")` when an active UI context exists, with stderr as the fallback for non-UI callers.

Warning delivery is still ad hoc across `pi-mux-subagents`: pane launch paths, orchestration tool calls, headless backend child runners, backend selection, and CLI utilities can each choose their own delivery mechanism. That makes future UI corruption regressions likely and leaves unclear when warnings should be visible to a human user, the parent model/caller, or only a command-line observer.

## Goal

Centralize user-visible runtime diagnostics and warning delivery behind a small typed diagnostics abstraction. Warning text should remain stable, but delivery should be selected by runtime context so TUI sessions use UI notifications, headless/CLI contexts preserve stderr observability, and orchestration calls can optionally expose structured warning metadata when the warning is relevant to the caller model.

## Scope

- Add a diagnostics helper module with a typed warning sink / diagnostics sink abstraction.
- Route existing user-visible runtime warnings through that abstraction instead of direct ad hoc `process.stderr.write()` calls where the warning can occur during subagent launch or orchestration.
- Support context-aware delivery:
  - Interactive pane/TUI sessions: deliver warnings through `ctx.ui.notify(message, "warning")` when available.
  - Non-UI pane launches: fall back to stderr.
  - Headless backend child runners: preserve stderr for process-level diagnostics and command-line observability.
  - CLI utilities such as `pi-mux-detect`: preserve stderr behavior.
  - Orchestration tool calls: include narrow structured warning metadata/details only when the warning is relevant to the parent model/caller.
- Keep existing warning wording byte-for-byte where tests or downstream users may depend on it.
- Add regressions for pane/TUI warning routing, stderr fallback, headless stderr behavior, and structured metadata in the narrow caller-relevant cases.
- Do not rework general child-process stderr capture, spawned-tool process errors, or CLI warning wording unless a call site must move through the diagnostics abstraction.

## Acceptance Sketch

- A pane/TUI warning path test proves warnings are delivered via `ctx.ui.notify(message, "warning")` and do not write raw stderr when UI notification is available.
- A non-UI pane launch test proves the same warning text falls back to stderr unchanged.
- A headless backend warning test proves current stderr behavior is preserved.
- A CLI utility warning test proves CLI diagnostics still write to stderr.
- An orchestration tool result test proves caller-relevant warnings can be surfaced as structured metadata/details without changing user-facing warning text.
- Existing warning-string tests continue to pass byte-for-byte.
- New warning-producing call sites can be wired by selecting a diagnostics sink rather than deciding locally between `ctx.ui.notify`, structured metadata, and `process.stderr.write`.
