## Context

Pane-backed subagent launches can run inside Pi's interactive TUI. In that mode,
direct writes to `process.stderr` can appear as raw terminal output and corrupt
the active prompt editor or custom UI surfaces.

The immediate fix is to route pane launch warnings through `ctx.ui.notify()` when
an active UI context exists, with stderr as the fallback for non-UI callers.

## Follow-up Idea

Add one diagnostics abstraction for all user-visible runtime warnings emitted by
`pi-mux-subagents`.

The abstraction should keep warning text stable, but route delivery by context:

- Interactive pane/TUI sessions: `ctx.ui.notify(message, "warning")`.
- Orchestration tool calls: structured result or detail metadata when the warning
  is relevant to the caller model.
- Headless backend child runners: stderr remains acceptable for process-level
  diagnostics and command-line observability.
- CLI utilities such as `pi-mux-detect`: stderr remains appropriate.

## Why

This prevents future fixes from patching individual warning call sites after UI
corruption appears. It also makes warning behavior explicit for pane, headless,
and CLI contexts instead of relying on ad hoc `process.stderr.write()` calls.

## Possible Scope

1. Add a small `diagnostics` helper module with a typed warning sink.
2. Pass the sink into pane launch, headless backends, and backend selection paths.
3. Keep existing warning wording byte-for-byte where tests depend on it.
4. Add regressions for:
   - pane/TUI warnings never writing raw stderr,
   - non-UI pane launch warnings still falling back to stderr,
   - headless warnings preserving current stderr behavior,
   - structured warning metadata only where it is useful to the parent model.

## Out Of Scope

- Reworking all child process stderr capture.
- Changing warning wording or CLI behavior without a separate migration note.
- Treating process-level errors from spawned tools as UI notifications.
