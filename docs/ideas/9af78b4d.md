Add a direct way to focus an already-running pane-backed subagent, via a dedicated `subagent_focus({ id | name })` tool/command.

Current codebase facts:
- Launch-time `focus?: boolean` already exists on the shared subagent launch surface and is routed through bare `subagent`, `subagent_resume`, `subagent_run_serial`, and `subagent_run_parallel` tasks.
- That launch-time option only affects whether a newly spawned pane grabs focus. In current mux code it is effectively tmux-only (`createSurface(..., { detach })`); other mux backends may still focus new panes as a documented limitation.
- `pi-extension/subagents/index.ts` already tracks currently running subagents in the module-local `runningSubagents` map. `RunningSubagent` has stable `id`, display `name`, `backend`, and pane-only `surface?: string` fields.
- Pane launches populate `backend: "pane"` and `surface`; headless launches populate `backend: "headless"` and have no surface.
- Virtual blocked orchestration rows can appear in `runningSubagents` with `backend: "pane"` but no real `surface`; they must not be focusable.
- `pi-extension/subagents/cmux.ts` currently has mux helpers for creating, reading, sending to, and closing surfaces, but no `focusSurface(surface)` helper.
- Widgets currently do not receive keyboard focus/input, so a direct "select from widget" interaction is not feasible with the current pi widget API. Widget integration should be limited to displaying enough information, or deferred until the API changes.

Typical use case:
- A user launches `subagent_run_parallel(...)` with 4-8 pane-backed tasks.
- One task looks stuck or especially interesting in the widget/status output.
- Instead of remembering mux-specific navigation shortcuts, the user calls something like `subagent_focus({ name: "task-2" })` or `subagent_focus({ id: "..." })` and jumps directly to the correct pane.

Why this matters:
- Speeds up the loop from overview -> inspect/intervene.
- Improves usability for users who are not fluent in tmux/zellij/wezterm/cmux navigation.

Suggested implementation shape:
1. Add a mux helper such as `focusSurface(surface: string): void` in `pi-extension/subagents/cmux.ts`.
   - Implement backend-specific focus only where the mux supports targeting an existing pane/surface.
   - Return/throw a clear unsupported-backend error for muxes where arbitrary pane focusing is not available or not yet implemented.
2. Register a `subagent_focus` tool/command from the subagents extension.
   - Parameters: require exactly one of `id` or `name`.
   - Prefer `id` as the stable, unambiguous selector.
   - Allow `name` only when it matches exactly one currently running focusable subagent.
3. Resolve focus candidates from `runningSubagents`.
   - Focusable means `backend === "pane"`, `surface` is present, and the row is not virtual/blocked.
   - Headless subagents, completed subagents, and blocked virtual rows should fail with clear messages.
   - Ambiguous names should fail and list matching ids/names so the caller can retry by id.
4. Consider making ids easier to discover.
   - The widget currently renders names, not ids. Either include a short id in the widget/result details, or have ambiguous/not-found errors list current focusable candidates.
5. Add tests.
   - Unit-test selector resolution: id match, unique name match, ambiguous name, missing target, headless target, blocked virtual row/no surface.
   - Unit-test `focusSurface` command construction or unsupported errors using the existing mux/test seam style.
   - Integration-test tool registration/execution with mocked running entries and mocked focus helper.

Success criteria:
- Users can reliably jump to a running pane-backed subagent by stable id, and by name when unambiguous.
- Missing, ambiguous, completed, headless, blocked, and unsupported-backend cases fail with clear actionable messages.
- Backend-specific behavior and limitations are documented in README/tool help.
