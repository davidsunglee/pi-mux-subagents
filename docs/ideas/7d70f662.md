## Context
Claude CLI subagent dispatch currently bypasses Claude's tool permission checks by default. Pane launches use `--dangerously-skip-permissions`, and headless launches use `--permission-mode bypassPermissions`. This makes autonomous subagent runs smoother, but it removes Claude's permission classifier from decisions about risky actions.

The safer default should not be modeled as a Claude-specific agent setting. Agent configuration should stay mostly CLI-agnostic because future subagent backends may include Codex, OpenCode, and pi itself, each with different safety controls:

- Claude has a direct guarded mode via `--permission-mode auto`, and an unrestricted/bypass mode via `--permission-mode bypassPermissions` or the legacy `--dangerously-skip-permissions` path.
- Codex has a logical guarded equivalent through `--sandbox workspace-write --ask-for-approval on-request`, with `approvals_reviewer=auto_review` as the closest AFK/non-interactive equivalent to classifier-backed approval review. Its unrestricted equivalent is `--dangerously-bypass-approvals-and-sandbox` or `--sandbox danger-full-access --ask-for-approval never`.
- OpenCode has permissions that resolve to `allow`, `ask`, or `deny`, but no true classifier-backed `auto` equivalent. Its unrestricted equivalent is broadly `permission: "allow"`; a future guarded mapping would need to be conservative/default-policy-based rather than equivalent to Claude auto.
- pi currently has no comparable permission classifier or sandbox policy for tool execution. In this policy dimension, existing pi subagent behavior is closest to unrestricted/YOLO, subject only to existing tool availability and deny-tool configuration.

## Goal
Change Claude CLI subagent dispatch so guarded execution is the default for Claude, while introducing a CLI-agnostic policy concept that can map cleanly across current and future CLI backends.

Use the finalized generic launch/agent option names:

- API/tool parameter: `executionPolicy`
- Agent frontmatter: `execution-policy`

Supported initial values:

- `guarded`: prefer the backend's safest practical autonomous mode.
- `unrestricted`: explicitly opt into bypass/full-access behavior for trusted, sandboxed, or otherwise controlled runs.

For this idea, use fastlane scope and implement the behavior for Claude only. Document the intended future mappings for Codex, OpenCode, and pi without requiring full support for those backends in the same change.

## Scope
- Add a CLI-agnostic execution policy surface rather than adding Claude-specific `permissionMode` frontmatter.
- Apply the same policy surface to `subagent`, `subagent_run_serial`, and `subagent_run_parallel` so direct and orchestrated launches behave consistently.
- Add `execution-policy` frontmatter parsing using the existing kebab-case frontmatter convention.
- For Claude pane launches, map the default/`guarded` policy to `--permission-mode auto` instead of `--dangerously-skip-permissions`.
- For Claude headless launches, map the default/`guarded` policy to `--permission-mode auto` instead of `--permission-mode bypassPermissions`.
- For Claude launches with `executionPolicy: "unrestricted"` or `execution-policy: unrestricted`, preserve an explicit escape hatch that restores bypass behavior using the appropriate Claude CLI argument for the launch path.
- For backends without implemented guarded support, warn and continue with documented current behavior rather than rejecting the launch.
- Keep `CLAUDE_CODE_SANDBOXED=1` for pane launches unless a better non-interactive workspace-trust solution is identified; this bypasses the trust dialog, not tool permissions.
- Document future backend mappings: Codex guarded/unrestricted, OpenCode best-effort guarded/unrestricted, and pi's current unrestricted-only behavior.
- Update tests, fixtures, or workflows that assume Claude bypass-by-default.

Out of scope:
- Removing bypass/unrestricted mode entirely.
- Implementing Codex or OpenCode dispatch in this change.
- Adding a real permission classifier or sandbox layer to pi.
- Solving every possible Claude refusal caused by guarded execution.
- Running a full spec/plan/execute cycle; fastlane is sufficient for this bounded Claude-first change.

## Acceptance Sketch
- Claude pane subagent launches default to `--permission-mode auto` and no longer pass `--dangerously-skip-permissions` by default.
- Claude headless subagent launches default to `--permission-mode auto` and no longer pass `--permission-mode bypassPermissions` by default.
- A caller can explicitly request `executionPolicy: "unrestricted"` to restore Claude bypass behavior for trusted/sandboxed runs.
- Direct subagent launches and serial/parallel orchestration tasks expose the same `executionPolicy` option and resolve it consistently.
- Agent frontmatter uses `execution-policy`, not Claude-specific `permission-mode`.
- Existing non-permission-related launch behavior remains unchanged, including pane sandbox/trust handling unless intentionally revised.
- Unsupported/non-Claude backends warn and continue with documented current behavior when guarded behavior is requested but not implemented.
- Tests or verification cover generated Claude CLI arguments for guarded and unrestricted modes across pane and headless dispatch paths.
- Documentation explains the default change, the unrestricted escape hatch, and the non-equivalent future mappings for Codex, OpenCode, and pi.
- Migration notes call out that workflows relying on bypass-by-default may now see Claude refuse or avoid risky actions such as destructive git operations, credential exploration, production access, or irreversible deletes.

## Open Questions
- Should Codex guarded mode require `approvals_reviewer=auto_review` for all non-interactive runs, or should that be an additional stricter preset layered on top of `workspace-write` plus `on-request` when Codex support is implemented?
- What exact best-effort OpenCode guarded permission profile should be used if OpenCode backend support is added later?
- Should pi eventually gain a real guarded execution mode, or should it remain documented as unrestricted-only until a separate sandbox/permission feature exists?
