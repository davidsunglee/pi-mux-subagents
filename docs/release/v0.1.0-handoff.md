# v0.1.0 fresh-repo migration and publish handoff

This runbook records the maintainer-owned migration from the verified `pi-interactive-subagent` working tree into the fresh GitHub repo `pi-mux-subagents`, then the first public npm publish of `@aphotic/pi-mux-subagents@0.1.0`.

## prerequisites

Before running any migration or publish command:

1. Tasks 1–11 from `docs/plans/2026-05-20-a1503e6a.md` have passed in the source repo.
2. `pnpm run check` exits 0 in the source repo.
3. The fresh GitHub repo `pi-mux-subagents` already exists and was created manually by the maintainer, not as a GitHub fork.
4. The fresh repo is bootstrapped only with `.gitignore` and `README.md`.
5. `package.json` in this source tree already contains the final repository URL, with no placeholder brackets.
6. The maintainer is authenticated to npm under an account that can publish packages in the `@aphotic` scope.
7. pnpm 11.1.3 is installed locally (matching the repo's `packageManager` field) and available on `PATH`.

Spec release mechanics, reproduced verbatim:

> - All refactor and verification work happens in the existing `pi-interactive-subagent` repo first, where tests pass and history is available.
> - The maintainer creates a fresh GitHub repo named `pi-mux-subagents` manually (not a GitHub fork), bootstrapped with only `.gitignore` and `README.md`.
> - The cleaned working tree is then copied into the fresh repo without `.git/`.
> - No "Initial import from upstream" attribution commit is required; the README's Attribution section is the canonical attribution surface.
> - The first git tag in the fresh repo is `v0.1.0`, placed at the initial publish-ready commit.
> - `npm publish --access public` runs from the fresh repo (required because the package name is scoped).

## working-tree copy

Clone the fresh repo into a temporary publish directory, then copy the cleaned source tree into it. The copy excludes only local/generated artifacts and the source repo's git history. The npm tarball whitelist (`files: ["src", "README.md", "LICENSE"]`) is what keeps `docs/`, `test/`, `tsconfig.json`, and other contributor-only files out of the published package; the fresh source repo keeps those files so contributors can run tests and read this runbook. `pnpm-lock.yaml` and `pnpm-workspace.yaml` are included for reproducible installs in CI.

```sh
git clone "$(node -e "console.log(require('./package.json').repository.url)")" /tmp/pi-mux-subagents-publish
rsync -av --exclude='.git/' --exclude='.worktrees/' --exclude='node_modules/' \
  --exclude='lib/' \
  ./ /tmp/pi-mux-subagents-publish/
```

`.pi/` is intentionally not excluded. Its tracked subdirectories (`.pi/skills`, `.pi/settings`, etc.) are part of the maintainer workflow in the fresh repo. If the maintainer prefers a `git ls-files`-driven copy instead of `rsync`, that is acceptable as long as it produces the same cleaned tree and does not copy `.git/`, `.worktrees/`, `node_modules/`, or `lib/`.

After copying, confirm the destination still has its own `.git/` directory and contains the expected project files:

```sh
test -d /tmp/pi-mux-subagents-publish/.git
test -f /tmp/pi-mux-subagents-publish/src/index.ts
test -f /tmp/pi-mux-subagents-publish/package.json
test -f /tmp/pi-mux-subagents-publish/README.md
test -f /tmp/pi-mux-subagents-publish/LICENSE
```

## repository.url substitution

Before any commit, tag, push, or publish step, re-run the defensive placeholder check inside the fresh repo:

```sh
cd /tmp/pi-mux-subagents-publish
node -e "const p=require('./package.json'); const url=p.repository.url; if (url.includes('<') || url.includes('>')) { console.error('repository.url still contains placeholder brackets:', url); process.exit(1) }"
```

This command must exit 0 before `git add`, `git commit`, `git tag`, `git push`, or `npm publish --access public` runs. If it fails, substitute the real owner/organization slug in `package.json` and rerun the check until it exits 0.

## tag+publish

From the fresh repo, install dependencies, run the full verification, commit the first publish-ready snapshot, tag it, push, then publish.

```sh
cd /tmp/pi-mux-subagents-publish
pnpm install --frozen-lockfile
pnpm run check
node -e "const p=require('./package.json'); const url=p.repository.url; if (url.includes('<') || url.includes('>')) { console.error('repository.url still contains placeholder brackets:', url); process.exit(1) }"

git add . && git commit -m "Initial release of @aphotic/pi-mux-subagents 0.1.0"
git tag v0.1.0
git push origin main --tags
npm publish --access public
```

Dependency installation and verification use pnpm (matching the repo's `packageManager` field and tracked `pnpm-lock.yaml`). The publish step itself stays on npm because `npm publish` is the canonical mechanism for the npm registry; pnpm can publish too, but npm is the recorded contract in the spec.

`npm publish --access public` is required because this is the first publish of a scoped package. `npm publish` will run `prepublishOnly` or `prepublish` lifecycle hooks if they exist; none currently exist and this release does not add them.

Do not proceed to commit/tag/publish if `pnpm run check` fails. Debug in the source repo, recopy the cleaned tree, and rerun verification in the fresh repo.

## post-publish smoke check

After publish, verify the public registry metadata and a clean consumer install:

```sh
npm view @aphotic/pi-mux-subagents@0.1.0 --json > /tmp/published.json
node -e "const v=require('/tmp/published.json'); process.exit(v.name === '@aphotic/pi-mux-subagents' && v.version === '0.1.0' && v.repository && typeof v.repository.url === 'string' && v.repository.url.endsWith('/pi-mux-subagents') ? 0 : 1)"

consumer_dir=$(mktemp -d /tmp/pi-mux-consumer-XXXXXX)
cd "$consumer_dir"
npm init -y >/dev/null
npm install @aphotic/pi-mux-subagents@0.1.0
./node_modules/.bin/pi-mux-detect | node -e "const r=JSON.parse(require('fs').readFileSync(0,'utf8')); for (const k of ['backend','mux','modeForced','muxPreference','muxPreferenceInvalid','reason']) if (!(k in r)) process.exit(1)"
```

Save the publish, `npm view`, and clean-install outputs to `/tmp/pi-mux-subagents-0.1.0-publish.log` for the maintainer's audit trail. Do not commit that local log.

## out-of-scope

Spec non-goals, reproduced verbatim:

> - Decomposing the ~2771-line `src/index.ts` into smaller modules. Deferred to a follow-up release (potentially `0.2.0`).
> - Implementing actual integration with additional CLIs (codex, opencode). The README mentions these as future scope; no code is added to support them in `0.1.0`.
> - Shipping any bundled agent definitions. The package's `agents/` directory does not exist in the published tarball. Consumers register their own agents or pull them from another package (e.g., pi-flow-core).
> - Exhaustive API documentation. The README's Quickstart is the entire documentation surface for `0.1.0`; per-backend, per-mux-adapter, per-export type docs are out of scope.
> - Adding a `main` or `exports` field. The package remains pi-CLI-loaded only, plus the explicit mux-detection `bin` command.
> - Backwards compatibility with the previously-declared `@davidsunglee/pi-interactive-subagent` name. That name has never been published, so no consumer migration path is required.
> - Publishing any subset of `test/` to the npm tarball (including integration agent fixtures like `test/integration/agents/*.md`).
> - Decisions about the disposition of the existing `pi-interactive-subagent` GitHub repo after migration (archiving, deleting, leaving as-is). The maintainer handles this outside the scope of this spec.
> - Modifying the separate `pi-flow` repository's `define-spec/detect-mux-backend` helper. This spec publishes the reusable detection command and verifies its contract; switching `pi-flow` over to that command is a downstream follow-up task.
> - Migrating the existing `4.0.0` version tag or any prior commits into the fresh repo.
> - Re-organizing files beyond the directory-shape rename described under "Source tree restructure (aggressive)". The new `src/tools/` and `src/launch/` groupings are populated only with existing files; no file content is split or merged.
