#### Minor (Nice to Have)

- **src/backends/codex-stream.ts:57: inline `projects={…}` override replaces the entire projects table for the launch**
  - **What:** `-c projects={"<cwd>"={trust_level="trusted"}}` sets the whole `projects` key, shadowing any project-trust entries from the user's `~/.codex/config.toml` for that launch (the previous dotted form merged a single key).
  - **Why it matters:** Harmless for single-cwd subagents (only the launch cwd's trust is consulted), but if a future Codex version consults trust for other directories mid-session, those would appear untrusted. Also, the fix is calibrated to observed Codex CLI 0.138 behavior; a later Codex release may fix dotted-key observation or change inline-table handling.
  - **Recommendation:** Keep the existing comment (it is good), and add a short note to the release/upgrade checklist to re-verify the trust bypass when bumping the supported Codex CLI version.
- **src/mux/adapters/herdr.ts:485-499: compaction retry applied only to `sendCommand`**
  - **What:** `sendEscape`, `readScreen`, and `readScreenAsync` still resolve the surface once with no retry; the same public-id compaction race that motivated the retry wrapper can hit them (e.g. `sendEscape` during interrupt, `readScreen` inside screen-polling loops).
  - **Why it matters:** A compaction at the wrong moment fails an interrupt or a single read with `pane_not_found` instead of transparently re-resolving. Read paths are usually polled so the impact is low, which is why this is Minor.
  - **Recommendation:** Wrap the remaining pane actions in `runHerdrPaneActionWithRetry` in a follow-up (read paths may want fewer attempts), or document why `sendCommand` is the only path that needs it.
- **src/mux/adapters/herdr.ts:463-483: partial-failure mode of the two-phase send**
  - **What:** If `send-text` succeeds but the `Enter` phase exhausts its retries (pane closed permanently between phases), the command is left typed-but-unsubmitted in the pane, and the thrown error doesn't distinguish "nothing sent" from "text sent, Enter failed". Worst-case added latency is also ~1s of synchronous `Atomics.wait` blocking (2 phases × 5 attempts × 100ms) on the parent's event loop.
  - **Why it matters:** A caller that catches the error and retries `sendCommand` would double-type the command text. Today no caller does this, and a permanently-closed pane fails the launch anyway, so the practical risk is low.
  - **Recommendation:** Consider including the failed phase in the error message; revisit only if a retry-at-caller pattern emerges.
- **src/backends/headless.ts:497-535: residual duplication window in the `agent_end` backfill**
  - **What:** The suffix/prefix overlap reconciliation assumes missed live-stream messages are a tail. If a mid-stream assistant `message_end` is lost AND the corresponding `agent_end` copies lack stable keys (no responseId/timestamp), the overlap computes 0 and content-key dedupe is the only guard, which intentionally permits repeated identical content — so a duplicate could slip through.
  - **Why it matters:** Requires a double failure (mid-stream loss + metadata loss) that the existing tests show was never observed; the realistic loss mode (tail loss near process exit) is fully covered by `headless-pi-agent-end.test.ts`.
  - **Recommendation:** No change needed for this release; note the assumption ("losses are tails") in the comment block if this code is touched again.
- **.pi/skills/herdr/SKILL.md (untracked): provenance unrecorded**
  - **What:** The skill reads like a condensed copy of herdr's official agent documentation (it links to herdr.dev docs) and carries no provenance or attribution note. It is untracked, project-local, and excluded from the npm package by the `files` whitelist, so it has no packaging or license impact on the v0.3.0 artifact.
  - **Why it matters:** If it is later committed, the repo should record where it came from and under what terms, since the repo itself is MIT-licensed.
